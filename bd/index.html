Bases de Datos

Un viaje completo desde los archivos hasta el modelo físico

¿Por qué NO usar archivos?

Redundancia de Datos

La misma información se repite en múltiples lugares, desperdiciando espacio y causando inconsistencias.

Falta de Integridad

No hay mecanismos para garantizar que los datos sean válidos y consistentes.

Concurrencia Limitada

Múltiples usuarios no pueden acceder simultáneamente sin riesgo de corrupción de datos.

Consultas Ineficientes

Buscar información requiere leer todo el archivo secuencialmente.

Ejemplo: Problema con Archivos
// Archivo: estudiantes.txt Juan Pérez, Ingeniería, Prof. García María López, Ingeniería, Prof. García // Redundancia del profesor Carlos Ruiz, Medicina, Dra. Torres // Problemas: // 1. ¿Qué pasa si cambio el nombre del Prof. García? // 2. ¿Cómo busco todos los estudiantes de Ingeniería? // 3. ¿Qué pasa si dos personas modifican a la vez?

Conceptos Fundamentales de Bases de Datos

¿Qué es una Base de Datos?

Una base de datos es una colección organizada de datos estructurados que se almacenan electrónicamente en un sistema informático, diseñada para ser accedida, gestionada y actualizada de manera eficiente.

Sistema de Gestión de Base de Datos (SGBD)

Un SGBD es el software que permite crear, mantener y manipular bases de datos. Ejemplos: MySQL, PostgreSQL, Oracle, SQL Server, MongoDB.

Seguridad

Control de acceso y permisos

Transacciones

ACID: Atomicidad, Consistencia, Aislamiento, Durabilidad

Respaldo

Backup y recuperación

Conceptos Clave

1. Datos vs Información

Datos: Hechos crudos (Ej: "25", "Juan", "2024")

Información: Datos procesados con significado (Ej: "Juan tiene 25 años en 2024")

3. Abstracción de Datos
  • Nivel Físico: Cómo se almacenan los datos
  • Nivel Lógico: Qué datos se almacenan
  • Nivel de Vista: Cómo los usuarios ven los datos
5. Independencia de Datos

Física: Cambios en el almacenamiento no afectan las aplicaciones

Lógica: Cambios en la estructura no afectan las aplicaciones

2. Esquema vs Instancia

Esquema: Estructura de la BD (diseño)

Instancia: Datos actuales en un momento dado

4. Integridad de Datos

Entidad: Cada registro debe ser único

Referencial: Las referencias entre tablas deben ser válidas

Dominio: Los valores deben estar en rangos permitidos

6. Normalización

Proceso de organizar datos para reducir redundancia y mejorar integridad (1NF, 2NF, 3NF, BCNF)

Modelo Entidad-Relación (MER)

¿Qué es el MER?

El Modelo Entidad-Relación es una herramienta de diseño conceptual que representa la estructura de una base de datos de forma visual mediante entidades, atributos y relaciones.

Componentes del MER

Entidades

Objetos o conceptos del mundo real

Ej: Estudiante, Profesor, Curso

Atributos

Características de las entidades

Ej: nombre, edad, dirección

Relaciones

Asociaciones entre entidades

Ej: estudia, enseña, inscribe

Tipos de Atributos
Clave: Identificador único
Simple: No divisible (Ej: edad)
Compuesto: Divisible (Ej: dirección)
Multivaluado: Múltiples valores (Ej: teléfonos)
Derivado: Calculado (Ej: edad desde fecha_nacimiento)
Opcional: Puede ser NULL
Cardinalidad de Relaciones
Uno a Uno (1:1)

Un estudiante tiene un único carnet

Uno a Muchos (1:N)

Un profesor enseña muchos cursos

Muchos a Muchos (N:M)

Estudiantes inscriben múltiples cursos

Ejemplo Completo: Sistema Universitario
ENTIDADES: ┌─────────────┐ ┌─────────────┐ ┌─────────────┐ │ ESTUDIANTE │ │ CURSO │ │ PROFESOR │ ├─────────────┤ ├─────────────┤ ├─────────────┤ │ *id_est │ │ *id_curso │ │ *id_prof │ │ nombre │ │ nombre │ │ nombre │ │ apellido │ │ creditos │ │ titulo │ │ email │ │ nivel │ │ email │ │ f_nacim │ │ descripcion│ │ salario │ └─────────────┘ └─────────────┘ └─────────────┘ RELACIONES: - ESTUDIANTE (N) --- INSCRIBE --- (M) CURSO Atributos: fecha_inscripcion, calificacion - PROFESOR (1) --- ENSEÑA --- (N) CURSO Atributos: año, semestre * = Clave primaria

Modelo Relacional (MR)

¿Qué es el Modelo Relacional?

El Modelo Relacional representa los datos como tablas (relaciones) con filas (tuplas) y columnas (atributos). Fue propuesto por Edgar F. Codd en 1970.

Componentes del Modelo Relacional

Relación (Tabla)

Conjunto de tuplas con la misma estructura

  • Grado: Número de atributos
  • Cardinalidad: Número de tuplas
Tupla (Fila)

Cada registro de datos en la tabla

Ejemplo: (101, "Juan", "Pérez", 20)

Atributo (Columna)

Cada campo con un nombre y tipo de datos

Ejemplo: nombre VARCHAR(50)

Dominio

Conjunto de valores válidos para un atributo

Ejemplo: edad ∈ {0, 1, 2, ..., 120}

Claves en el Modelo Relacional

Clave Primaria (PK)

Atributo(s) que identifican únicamente cada tupla

Propiedades:

  • Única
  • No nula
  • Inmutable
Clave Foránea (FK)

Atributo que referencia la PK de otra tabla

Propósito:

  • Establecer relaciones
  • Mantener integridad referencial
Clave Candidata

Cualquier atributo que podría ser PK

Ej: email, DNI, matrícula

Clave Compuesta

PK formada por múltiples atributos

Ej: (id_estudiante, id_curso)

Conversión de MER a MR
Reglas de Transformación:
  1. Entidades: Se convierten en tablas
  2. Atributos simples: Se convierten en columnas
  3. Atributos compuestos: Se descomponen en atributos simples
  4. Atributos multivaluados: Se crean tablas separadas
  5. Relaciones 1:1: FK en cualquiera de las dos tablas
  6. Relaciones 1:N: FK en el lado N
  7. Relaciones N:M: Se crea tabla intermedia con las PKs como FKs
Ejemplo: Transformación a Tablas
-- TABLA: Estudiante CREATE TABLE Estudiante ( id_estudiante INT PRIMARY KEY, nombre VARCHAR(50) NOT NULL, apellido VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE, fecha_nacimiento DATE ); -- TABLA: Curso CREATE TABLE Curso ( id_curso INT PRIMARY KEY, nombre VARCHAR(100) NOT NULL, creditos INT CHECK (creditos > 0), id_profesor INT, FOREIGN KEY (id_profesor) REFERENCES Profesor(id_profesor) ); -- TABLA: Inscripcion (N:M) CREATE TABLE Inscripcion ( id_estudiante INT, id_curso INT, fecha_inscripcion DATE, calificacion DECIMAL(4,2), PRIMARY KEY (id_estudiante, id_curso), FOREIGN KEY (id_estudiante) REFERENCES Estudiante(id_estudiante), FOREIGN KEY (id_curso) REFERENCES Curso(id_curso) );

Modelo Físico

¿Qué es el Modelo Físico?

El Modelo Físico describe cómo se implementan y almacenan los datos en el hardware. Define estructuras de almacenamiento, índices, particiones y optimizaciones específicas del SGBD.

Componentes del Modelo Físico

Tablespaces

Espacios de almacenamiento donde se guardan los datos físicamente

Índices

Estructuras que aceleran las búsquedas y consultas

Particiones

División de tablas grandes en piezas más manejables

Tipos de Índices
Índice B-Tree

Estructura de árbol balanceado, ideal para búsquedas por rangos

CREATE INDEX idx_estudiante_apellido ON Estudiante(apellido);
Índice Hash

Tabla hash, muy rápido para búsquedas exactas

CREATE INDEX idx_curso_codigo ON Curso(codigo) USING HASH;
Índice Compuesto

Índice sobre múltiples columnas

CREATE INDEX idx_inscripcion ON Inscripcion(id_estudiante, id_curso);
Índice Único

Garantiza valores únicos

CREATE UNIQUE INDEX idx_email ON Estudiante(email);
Optimización Física
Consideraciones de Diseño Físico:
Desnormalización: Agregar redundancia para mejorar rendimiento
Clustering: Organizar datos físicamente según acceso frecuente
Compresión: Reducir espacio de almacenamiento
Buffering: Uso de memoria cache
Estadísticas: Información para optimizador de consultas
Particionamiento: Horizontal o vertical
Ejemplo Completo: Implementación Física
-- CREACIÓN DE TABLESPACE CREATE TABLESPACE ts_universidad DATAFILE 'universidad.dbf' SIZE 500M AUTOEXTEND ON NEXT 100M MAXSIZE 2G; -- TABLA CON ESPECIFICACIONES FÍSICAS CREATE TABLE Estudiante ( id_estudiante INT PRIMARY KEY, nombre VARCHAR(50) NOT NULL, apellido VARCHAR(50) NOT NULL, email VARCHAR(100) UNIQUE, fecha_nacimiento DATE, ciudad VARCHAR(50) ) TABLESPACE ts_universidad; -- ÍNDICES PARA OPTIMIZACIÓN CREATE INDEX idx_est_apellido ON Estudiante(apellido); CREATE INDEX idx_est_ciudad ON Estudiante(ciudad); -- PARTICIONAMIENTO POR RANGO CREATE TABLE Inscripcion ( id_inscripcion INT, id_estudiante INT, id_curso INT, fecha_inscripcion DATE, calificacion DECIMAL(4,2) ) PARTITION BY RANGE (fecha_inscripcion) ( PARTITION p_2022 VALUES LESS THAN ('2023-01-01'), PARTITION p_2023 VALUES LESS THAN ('2024-01-01'), PARTITION p_2024 VALUES LESS THAN ('2025-01-01') ); -- ESTADÍSTICAS PARA OPTIMIZADOR ANALYZE TABLE Estudiante COMPUTE STATISTICS; ANALYZE TABLE Inscripcion COMPUTE STATISTICS;

Resumen Final

¡Conceptos Clave Aprendidos!

Archivos vs Bases de Datos

Los archivos presentan problemas de redundancia, inconsistencia, concurrencia y dificultad para consultas. Las bases de datos solucionan estos problemas mediante SGBDs.

Modelo Entidad-Relación

Representación conceptual mediante entidades, atributos y relaciones. Define ESTRUCTURA y SIGNIFICADO sin implementación.

Modelo Físico

Implementación real con índices, particiones, tablespaces. Optimización del RENDIMIENTO y ALMACENAMIENTO.

Conceptos Fundamentales

SGBD, ACID, integridad, independencia de datos, abstracción de datos, normalización y transacciones.

Modelo Relacional

Implementación lógica mediante tablas con claves primarias y foráneas. Transformación del MER a estructuras relacionales.

Flujo de Diseño

MER (Conceptual) → MR (Lógico) → Físico (Implementación). Cada nivel agrega más detalles técnicos.
