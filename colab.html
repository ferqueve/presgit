Ventajas

  • Historial limpio y organizado
  • Fácil reversión de cambios
  • Testing independiente
Flujo típico

  • Desarrollar la funcionalidad
  • Hacer commits regulares

Configuración Inicial

Clonar el Repositorio
# Ana y Carlos clonan el repo
git clone https://github.com/empresa/login-app.git
cd login-app
Configurar Git
git config user.name "Tu Nombre"
git config user.email "tu@email.com"
Ver Branches Existentes
git branch -a
# Muestra:
* main
origin/main
Actualizar desde Remoto
git fetch origin
git pull origin main
Estado Inicial

Ambos desarrolladores tienen la misma versión del código en la rama main. Ahora cada uno creará su propia branch para trabajar de forma independiente.

Buenas Prácticas
  • Commits frecuentes: Pequeños cambios, descripciones claras
  • Mensajes descriptivos: Usar formato convencional (feat:, fix:, docs:)
  • Testing: Probar cada funcionalidad antes de commit
  • Sincronización: Pull de main regularmente para evitar conflictos grandes

Creando Pull Requests

¿Qué es un Pull Request?

Un Pull Request (PR) es una solicitud para fusionar cambios de una branch a otra (normalmente a main). Permite revisión de código y discusión antes de la fusión.

1
Ana crea su PR
  • Va a GitHub.com
  • Click en "Compare & pull request"
  • Título: "Add login UI components"
  • Descripción detallada de cambios
  • Asigna a Carlos como reviewer
2
Carlos crea su PR
  • Mismo proceso en GitHub
  • Título: "Implement authentication API"
  • Documenta endpoints y uso
  • Asigna a Ana como reviewer
Proceso de Review
  • Code Review: Revisar código línea por línea
  • Tests: Verificar que los tests pasan
  • Comentarios: Sugerir mejoras o hacer preguntas
  • Aprobación: Aprobar o solicitar cambios

¿Qué pasa si hay conflictos?

Escenario: Archivo Compartido

Imagina que tanto Ana como Carlos modificaron el archivo package.json para agregar dependencias diferentes.

1
Actualizar desde main antes de merge
# Ana actualiza su branch desde main
git checkout feature/login-ui
git fetch origin
git merge origin/main
2
Si hay conflictos, Git te avisa
Auto-merging package.json
CONFLICT (content): Merge conflict in package.json
Automatic merge failed; fix conflicts and then commit the result.
3
Resolver en VSCode

VSCode muestra los conflictos visualmente:

  • HEAD (tu branch): Tus cambios
  • Incoming: Cambios de main
  • Opciones: Accept Current, Accept Incoming, Accept Both
  • Editar manualmente si es necesario

Limpieza y Buenas Prácticas

Limpieza de Branches
# Eliminar branches locales
git branch -d feature/login-ui
git branch -d feature/auth-api

# Eliminar del remoto
git push origin --delete feature/login-ui
Mantener main actualizado
git checkout main
git pull origin main

# Ver branches remotas
git branch -r

# Limpiar referencias
git remote prune origin
Mejores Prácticas del Flujo
  • Nombres descriptivos: feature/login-ui, bugfix/auth-validation
  • Branches cortas: 1-3 días de trabajo máximo
  • Pull regular: Actualizar desde main frecuentemente
  • Tests: Verificar que todo funciona antes de PR
  • Reviews significativas: No solo "LGTM"
  • Documentación: PR descriptions detalladas
  • Comunicación: Avisar cambios importantes al equipo
  • Limpieza: Eliminar branches después del merge
