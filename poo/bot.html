El patrón

Façade: la puerta única a un subsistema

Un patrón estructural del catálogo de la Gang of Four. Provee una interfaz simple y unificada a un conjunto de clases más complejo. El cliente habla con la fachada; la fachada coordina por dentro.

Intención

Reducir el acoplamiento entre un cliente y un subsistema compuesto por muchas clases. El cliente no necesita conocer la estructura interna ni el orden en que se invocan las operaciones — solo conoce la fachada.

La metáfora

Pensá en la recepción de un hotel. No vas hasta lavandería para pedir toallas, ni hasta la cocina para que te suban un café, ni a la sala de máquinas si no funciona el aire. Vas a recepción. Recepción coordina con todos los servicios internos. Para vos, el hotel es una sola interfaz.

Antes de seguir: ¿qué NO es Façade?

No es ocultar un objeto detrás de otro objeto idéntico (eso es Proxy). No es agregar comportamiento dinámicamente (eso es Decorator). No es convertir una interfaz en otra incompatible (eso es Adapter). Façade simplifica el acceso a varios objetos coordinados — su valor no está en lo que hace, sino en lo que oculta.

El problema

Sin fachada: bot y dominio se enredan

Imaginá que el comando de Discord conoce directamente todas las clases del dominio. Cada comando del bot toca usuarios, roles y repositorios. Veámoslo:

UserInfoCommand.cs · versión SIN façade (hipotética)
public async Task ExecuteAsync(string displayName = null)
{
    var repository = new UsersRepository();
    User user = repository.Find(displayName);
    if (user == null) {
        await ReplyAsync("Usuario no existe");
        return;
    }
    string roles = string.Join(", ", user.Roles);
    await ReplyAsync($"Tiene roles: {roles}");
}
¿Qué se rompe acá?
  • El comando conoce UsersRepository, User y Role: tres clases.
  • Si la lógica de "qué decir cuando no existe" cambia, hay que tocar cada comando.
  • Probar la lógica requiere levantar Discord o mockear el contexto del bot.
  • Cada vez que se quiera agregar una funcionalidad, los detalles del dominio se repiten en otro comando.
Lo que queremos

Que el comando le diga a UN objeto: "¿qué información hay del usuario X?", reciba la respuesta lista, y la devuelva. Nada más.

Solución

Cuatro capas, una sola puerta

La demo organiza el código en cuatro capas. Las flechas marcan la dirección permitida de conocimiento — y notá que la fachada es el único punto por donde pasa todo.

La regla de oro de las capas

Commands NO conoce Domain. Solo conoce a Facade. Si mañana cambia User o UsersRepository, los comandos no se enteran. Y al revés: el dominio no sabe que existe Discord.

Capa Domain

Las clases del dominio: simples y encapsuladas

User

Tiene UserName y una colección privada de roles. Expone AddRole, RemoveRole, HasRole y un Roles de solo lectura.

private ICollection<Role> roles
    = new HashSet<Role>();
public IReadOnlyCollection<Role> Roles
Role

Solo tiene Name, validado en el constructor. Dos roles "comunes" pre-construidos como propiedades estáticas:

public static Role Admin { get; }
    = new Role("Admin");
public static Role User { get; }
    = new Role("User");
UsersRepository

Implementa IUsersRepository (interfaz). Guarda usuarios y los busca por nombre. La interfaz permite mockearlo en tests.

User Find(string userName);
IReadOnlyCollection<User> AllUsers
Detalle importante

Estas clases no saben que existe Discord, ni la fachada, ni nada externo. Son puro dominio: representan el negocio "usuarios con roles" y nada más. Esa es la condición que hace posible la fachada.

La Façade

Anatomía de la clase Facade

Facade.cs (resumido)
public class Facade
{
    private static Facade instance;

    // constructor privado: nadie crea instancias desde afuera
    private Facade()
    {
        this.usersRepository = new UsersRepository();
    }

    // constructor interno: para inyectar un mock en tests
    internal Facade(IUsersRepository repository)
    {
        ArgumentNullException.ThrowIfNull(repository);
        this.usersRepository = repository;
    }

    public static Facade Instance
    {
        get
        {
            if (instance == null)
            {
                instance = new Facade();
            }
            return instance;
        }
    }

    private IUsersRepository usersRepository;

    public string GetUserInfo(string userName) { … }
}
Singleton

Constructor privado + propiedad estática Instance. Solo existe una fachada en toda la aplicación. Los comandos siempre obtienen la misma.

El constructor internal

Existe específicamente para tests: permite construir una fachada con un IUsersRepository falso. Decisión de diseño consciente: el patrón Façade habilita el testing aislado.

Ojo con el singleton

Esta implementación NO es thread-safe (dos hilos podrían crear dos instancias en la primera invocación concurrente). Para el bot de la demo alcanza, pero es bueno saberlo.

La Façade

El método clave: GetUserInfo

Acá vive la única lógica que los comandos necesitan invocar para responder al usuario. Coordina con el repositorio y formatea el mensaje:

Facade.cs · método
public string GetUserInfo(string userName)
{
    string result;
    User userFound = this.usersRepository.Find(userName);

    if (userFound == null)
    {
        result = $"El usuario de Discord '{userName}' no es " +
                 $"usuario de esta aplicación.";
    }
    else
    {
        string roles = string.Join(", ", userFound.Roles);
        result = $"El usuario '{userName}' tiene los roles " +
                 $"'{roles}' en esta aplicación.";
    }

    return result;
}
  • Recibe un string, devuelve un string. Más simple imposible.
  • El cliente (el comando) nunca ve User ni Role: la fachada los maneja por dentro.
  • Si mañana cambiamos el mensaje, lo tocamos en un solo lugar.
  • Si cambia el dominio (otra forma de buscar usuarios), los comandos no se enteran.
Notá lo que está pasando

Esto es la fachada haciendo su trabajo: traduce una operación de negocio compleja ("buscar usuario, ver si existe, listar sus roles, formatear el resultado") en una llamada simple ("dame info de X").

Bootstrapping

El acertijo del primer administrador

Las historias de usuario dicen: "solo un admin puede crear otros usuarios". Pero entonces… ¿quién crea al primer admin? Es el problema clásico del huevo y la gallina.

UsersRepository.cs (extracto)
private void AddFirstAdmin()
{
    var configuration = new ConfigurationBuilder()
        .AddUserSecrets(Assembly.GetExecutingAssembly())
        .Build();

    string firstAdminName = configuration["FirstAdmin"];
    if (string.IsNullOrEmpty(firstAdminName))
    {
        throw new InvalidOperationException(
            "Falta el primer administrador.");
    }

    User firstAdmin = new User(firstAdminName);
    firstAdmin.AddRole(Role.Admin);
    firstAdmin.AddRole(Role.User);
    this.users.Add(firstAdmin);
}
La solución

El constructor del repositorio siembra al primer admin. Su nombre viene de secrets.json: un archivo local en cada máquina, fuera del repo de GitHub.

~/.microsoft/usersecrets/.../secrets.json
{
  "DiscordToken": "<token>",
  "FirstAdmin": "<usuario_discord>"
}
¿Por qué tanto trabajo?

Si el token y el nombre fueran a GitHub, cualquiera que viera el repo podría operar tu bot. Esta práctica de secretos por máquina escala bien: en producción se reemplaza por variables de entorno o un secret manager.

Capa Commands

El comando que TODO el bot necesita escribir

UserInfoCommand responde a !who [usuario]. Toda su conexión con el dominio es una sola línea. Buscala:

UserInfoCommand.cs
public class UserInfoCommand : ModuleBase<SocketCommandContext>
{
    [Command("who")]
    public async Task ExecuteAsync(
        [Remainder] string displayName = null)
    {
        if (displayName != null)
        {
            SocketGuildUser user =
                CommandHelper.GetUser(Context, displayName);
            if (user == null)
                await ReplyAsync($"No encuentro '{displayName}'");
        }

        string userName =
            displayName ?? CommandHelper.GetDisplayName(Context);

        string result = Facade.Instance.GetUserInfo(userName);

        await ReplyAsync(result);
    }
}
La línea resaltada

Es la única en todo el comando que toca el dominio. Todo lo demás es plomería de Discord: obtener el nombre, manejar el parámetro opcional, responder al canal.

¿Y para un comando nuevo?

Cualquier comando futuro (!adduser, !addrole, !list…) va a tener exactamente la misma estructura: extraer parámetros del contexto de Discord, llamar a un método de la fachada, devolver la respuesta. La fachada crece; los comandos son siempre delgados.

Un mensaje, de punta a punta

¿Notaste?

Las capas no se saltean. El comando no consulta al repositorio. La fachada no escribe en el canal. Cada una hace su parte.

El premio

Modo dual: la fachada vive sin Discord

El Program.cs de la demo prueba el punto en código: si recibe un argumento, invoca directamente a la fachada; si no, levanta el bot.

Program.cs
private static void Main(string [] args)
{
    if (args.Length != 0)
    {
        DemoFacade(args);   // sin bot
    }
    else
    {
        DemoBot();          // con bot
    }
}

private static void DemoFacade(string [] args)
{
    if (args.Length > 0)
    {
        Console.WriteLine(
            Facade.Instance.GetUserInfo(args[0]));
    }
}

private static void DemoBot()
{
    BotLoader.LoadAsync().GetAwaiter().GetResult();
}

La cita clave del teórico

"Es posible programar toda la funcionalidad pedida en la façade sin preocuparse por el bot de Discord, y agregar al final la funcionalidad del bot sin modificaciones a la façade."

Esto no es una promesa: es un hecho demostrable. Toda la suite de tests (FacadeTests) corre sin Discord, sin secrets.json, sin token. Solo con un mock del repositorio.

Testing

Por qué Façade es tan fácil de probar

El constructor internal y la interfaz IUsersRepository trabajan juntos: en el test inyectamos un mock y la fachada queda aislada.

FacadeTests.cs (extracto)
[Test]
public void GetUserInfo_WhenUserNotFound_ReturnsNotFoundMessage()
{
    Mock<IUsersRepository> mockedRepository;
    var facade = CreateFacadeWithMockRepo(out mockedRepository);
    mockedRepository.Setup(r => r.Find("Test")).Returns((User)null);

    var actual = facade.GetUserInfo("Test");

    Assert.That(actual, Is.EqualTo(
        "El usuario de Discord 'Test' no es usuario de esta aplicación."));
}
  • Sin Discord: el test no necesita un servidor ni un token.
  • Sin base de datos: el repositorio real (que lee secrets.json) ni se construye.
  • Determinístico: el mock responde exactamente lo que pedimos.
  • Rápido: milisegundos por test, no segundos.
El cierre del argumento

La fachada no es solo una "puerta": es un punto de testing y de evolución. Por eso ese constructor internal existe con todo el comentario explicativo.

Ejercicios desbloqueables

Construyamos un nuevo método de la fachada

Vas a extender la fachada con la operación "agregar un nuevo usuario común". El comando que la usa va a ser !adduser <nombre>. La cadena son 6 pasos: cada uno desbloquea al siguiente.

E1Detectar el patrón en una línea10 pts
E2Singleton: la línea de creación lazy10 pts
E3Firma del nuevo método en la fachada10 pts
E4El método Add en IUsersRepository10 pts
E5El cuerpo del nuevo método de la fachada10 pts
★Desafío: el comando !adduser que delega en la fachada20 pts
Reglas
  • Escribí solo lo que pide cada enunciado (una línea o una firma).
  • Las palabras clave de C# en minúscula (public, new, void); los tipos con la convención (User, Role).
  • El ; final es opcional.
  • Pista disponible: reduce el puntaje a la mitad.
Cadena · Parte 1

Ejercicios 1 a 3

E1 · DETECTIVE

En este comando hipotético hay UNA línea que respeta el patrón Façade (el comando le delega al dominio sin conocer sus detalles internos). Copiala tal cual aparece, sin la indentación.

public async Task ExecuteAsync(string name)
{
    var repo = new UsersRepository();
    User u = repo.Find(name);
    string result = Facade.Instance.GetUserInfo(name);
    await ReplyAsync(result);
}
E2 · SINGLETON LAZY

En la propiedad estática Instance, dentro del if (instance == null), escribí la línea que crea la instancia única usando el constructor privado.

E3 · FIRMA EN LA FAÇADE

Vamos a agregar un nuevo método a la fachada para que los comandos puedan crear usuarios. Escribí la firma del método AddUser: pública, sin retorno, recibe un string userName. (Solo la firma, sin el cuerpo.)

Cadena · Parte 2

Ejercicios 4 a 6

E4 · IUsersRepository.Add

Para que la fachada pueda agregar usuarios, el repositorio debe exponer un método. Declará en la interfaz IUsersRepository el método Add que recibe un User user y no retorna nada. (Solo la declaración, terminada en ;.)

E5 · CUERPO DEL MÉTODO

Dentro del método AddUser(string userName) de la fachada, ya validaste el parámetro. Ahora creá un nuevo User con el nombre recibido, asignalo a una variable local llamada newUser. (Una sola línea con la declaración, asignación y new.)

★ DESAFÍO · EL COMANDO !adduser

En el nuevo comando AddUserCommand, dentro de su ExecuteAsync(string name), escribí la única línea que delega en la fachada para crear el usuario. Recordá: el comando NO debe ver User ni UsersRepository, solo a Facade.

Cierre

Lo que te llevás de hoy

  • Façade es una puerta única a un subsistema: el cliente conoce poco, la fachada coordina mucho.
  • Cuatro capas en la demo: Discord → Commands → Façade → Domain.
  • El comando se conecta al dominio por una sola línea — todo lo demás es plomería.
  • El constructor internal + IUsersRepository habilitan testing aislado.
  • El modo dual del Program demuestra: la fachada vive sin Discord.
  • El primer admin sale de secrets.json: nunca al repo.
Para seguir

Cloná PII_DiscordBot_Demo, creá tu secrets.json con tu token y tu nombre de Discord, y probá:

$ dotnet run -- TuNombreDiscord

Después implementá !adduser con los ejercicios que acabás de hacer. Recordá: en las historias de usuario reales, solo un admin debería poder ejecutarlo — esa validación también va en la fachada.
