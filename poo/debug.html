Concepto Base

¿Qué es el Debug?

Depurar es pausar la ejecución del programa para inspeccionarlo por dentro

Sin Debug (antes)
Ponías Console.WriteLine() por todos lados para ver los valores. Lento, sucio, y tenías que borrarlo todo después.
Con Debug (ahora)
Pones un Breakpoint en la línea exacta, y VS Code pausa ahí para que puedas ver todas las variables, la pila de llamadas, y ejecutar línea por línea.

El depurador es el superpoder #1 de todo programador profesional.

1
Pones un Breakpoint
Hacés click en el margen izquierdo del editor, al lado del número de línea. Aparece un punto rojo.
2
Arrancás con F5
VS Code ejecuta el programa y lo pausa justo en tu breakpoint.
3
Inspeccionás y navegás
Ves los valores de las variables, avanzás línea por línea, y entendés exactamente qué pasa.
Dato histórico: El término "bug" viene de 1947, cuando Grace Hopper encontró una polilla real atrapada en un relé de la computadora Harvard Mark II causando un fallo. Ese fue literalmente el primer "bug" documentado.

TIPOS DE BREAKPOINTS
● Normal Breakpoint
Click en el margen izquierdo o F9 en la línea. Pausa siempre que llega a esa línea.
◆ Conditional Breakpoint
Click derecho en el punto rojo → "Edit Breakpoint". Solo pausa si se cumple una condición, ej: i == 5 o nombre == "Juan".
◎ Logpoint
Como un Console.WriteLine pero sin pausar. Imprime en la consola de Debug. Genial para loops sin interrumpir.
F9 — Toggle breakpoint en la línea actual  |  Ctrl+Shift+F9 — Borrar todos los breakpoints

Ejercicio Práctico 1

Debug en Acción

Ejercicio guiado: encontrá el bug en este código

Este programa debería calcular el promedio de una lista. Tiene un bug. Seguí los pasos:

Program.cs — código con bug 
using System;
using System.Collections.Generic;

List<int> numeros = new List<int> { 10, 20, 30, 40 };

double promedio = CalcularPromedio(numeros);
Console.WriteLine($"Promedio: {promedio}");

static double CalcularPromedio(List<int> lista)
{
    int suma = 0;
    for (int i = 0; i <= lista.Count; i++) // ← ¿ves el bug?
    {
        suma += lista[i];
    }
    return suma / lista.Count;
}
PASOS PARA DEBUGGEAR
1
Copiá el código en Program.cs
Creá un proyecto nuevo con dotnet new console y pegalo.
2
Pon breakpoint en la línea 12 (suma +=)
Hacé click en el margen izquierdo al lado del número 12. Aparece el punto rojo.
3
Presioná F5 para iniciar
El programa se pausa en la línea 12. Abrí el panel Variables a la izquierda.
4
Presioná F10 repetidas veces
Observá el valor de i en cada vuelta. ¿Qué pasa cuando i = 4?
5
¿Encontraste el bug?
El bug: i <= lista.Count debería ser i < lista.Count. Los índices van de 0 a 3, no a 4.

Test Unitario · Parte II

¿Qué es un Test Unitario?

Código que verifica que otro código funciona correctamente — automatizado

¿Qué prueba?
Una unidad = un método o función individual. Verificás que dado cierto input, el output sea el esperado.
¿Por qué automatizar?
Podés correr cientos de tests con un solo comando. Si algo se rompe, lo sabés al instante.
Arrange
Preparar los datos de entrada y el objeto a probar.
Act
Ejecutar el método que se está probando.
Assert
Verificar que el resultado sea el esperado.
Frameworks de Test en C#:   xUnit NUnit MSTest   En este curso usamos xUnit — el más popular en proyectos modernos de .NET.

namespace Calculadora;

public class OperacionesMatematicas
{
    public int Sumar(int a, int b)
    {
        return a + b;
    }

    public int Restar(int a, int b)
    {
        return a - b;
    }

    public double Dividir(int a, int b)
    {
        if (b == 0)
            throw new DivideByZeroException(
                "No se puede dividir por cero");
        return (double)a / b;
    }

    public bool EsPar(int numero)
    {
        return numero % 2 == 0;
    }
}
xUnit · Tests

Escribir los Tests

Cómo se escribe un test con xUnit en C# — patrón AAA

Calculadora.Tests/OperacionesTests.cs 
using Xunit;
using Calculadora;

namespace Calculadora.Tests;

public class OperacionesTests
{
    private readonly OperacionesMatematicas _calc;

    // Constructor: se ejecuta ANTES de cada test
    public OperacionesTests()
    {
        _calc = new OperacionesMatematicas();
    }

    [Fact]
    public void Sumar_DosNumeroPositivos_RetornaLaSuma()
    {
        // Arrange
        int a = 5, b = 3;

        // Act
        int resultado = _calc.Sumar(a, b);

        // Assert
        Assert.Equal(8, resultado);
    }

    [Fact]
    public void Dividir_PorCero_LanzaExcepcion()
    {
        // Assert + Act con excepción esperada
        Assert.Throws<DivideByZeroException>(
            () => _calc.Dividir(10, 0)
        );
    }

    [Theory]
    [InlineData(2, true)]
    [InlineData(3, false)]
    [InlineData(100, true)]
    [InlineData(7, false)]
    public void EsPar_Numeros_RetornaEsperado(
        int numero, bool esperado)
    {
        bool resultado = _calc.EsPar(numero);
        Assert.Equal(esperado, resultado);
    }
}
ATRIBUTOS CLAVE DE XUNIT
[Fact]
Un test simple, sin parámetros. Se ejecuta una sola vez. El más común.
[Theory] + [InlineData]
Test con múltiples conjuntos de datos. Se ejecuta una vez por cada [InlineData]. Ideal para probar varios casos sin repetir código.
ASSERTS MÁS USADOS
Métodos Assert de xUnit 
Assert.Equal(esperado, actual);
Assert.NotEqual(noEsperado, actual);
Assert.True(condicion);
Assert.False(condicion);
Assert.Null(objeto);
Assert.NotNull(objeto);
Assert.Throws<TipoExcepcion>(() => ...);
Assert.Contains(elemento, coleccion);

DEBUG DE UN TEST QUE FALLA
1
Instalá la extensión ".NET Core Test Explorer"
O usá el panel Testing nativo de VS Code (ícono del tubo de ensayo en la barra lateral).
2
Pon breakpoint dentro del test o la función
Igual que siempre — hacé click en el margen izquierdo en la línea que te interesa.
3
Click en "Debug Test" (sobre el [Fact])
VS Code muestra un lente 🔍 encima de cada [Fact]. Hacé click en "Debug" para arrancar el depurador sobre ese test específico.

Navegá con F10/F11 como siempre
El depurador funciona igual que con una aplicación normal. Ves todas las variables del test.
Cuando un test falla, el mensaje de error te dice el valor esperado vs el valor real. ¡Leélo primero antes de debuggear!
Ejercicio Final

Ejercicio Integrador

Debuggeá, testeá, y corregí — todo junto

Esta clase tiene 3 bugs. Tu misión: encontrarlos usando el debugger y corregirlos para que todos los tests pasen.

Banco.cs — código con 3 bugs 
namespace EjercicioFinal;

public class CuentaBancaria
{
    public decimal Saldo { get; private set; }

    public CuentaBancaria(decimal saldoInicial)
    {
        Saldo = saldoInicial;
    }

    // BUG #1: ¿cuál es?
    public void Depositar(decimal monto)
    {
        if (monto <= 0)
            throw new ArgumentException("Monto inválido");
        Saldo -= monto; // ← ¿bug acá?
    }

    // BUG #2: ¿cuál es?
    public void Retirar(decimal monto)
    {
        if (monto <= 0)
            throw new ArgumentException("Monto inválido");
        if (monto > Saldo)
            throw new InvalidOperationException("Saldo insuficiente");
        Saldo -= monto;
    }

    // BUG #3: ¿cuál es?
    public bool TieneSaldoSuficiente(decimal monto)
    {
        return Saldo > monto; // ← ¿bug acá?
    }
}
LOS TESTS (ya escritos)
CuentaBancariaTests.cs 
[Fact]
public void Depositar_MontoCorrecto_AumentaSaldo()
{
    var cuenta = new CuentaBancaria(100);
    cuenta.Depositar(50);
    Assert.Equal(150, cuenta.Saldo); // FALLA ← buscar bug
}

[Fact]
public void Retirar_SaldoExacto_SaldoQuedaEnCero()
{
    var cuenta = new CuentaBancaria(100);
    cuenta.Retirar(100);
    Assert.Equal(0, cuenta.Saldo); // PASA ✓
}

[Fact]
public void TieneSaldo_MontoIgualAlSaldo_RetornaTrue()
{
    var cuenta = new CuentaBancaria(100);
    bool resultado = cuenta.TieneSaldoSuficiente(100);
    Assert.True(resultado); // FALLA ← buscar bug
}
