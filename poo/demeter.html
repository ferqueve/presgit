Contexto

GRASP: asignar responsabilidades con criterio

Los patrones GRASP (General Responsibility Assignment Software Patterns) fueron planteados por Craig Larman en Applying UML and Patterns (1998). No son recetas de código: son criterios para decidir qué clase debe hacer qué cosa.

Expert Creator Controller Low Coupling High Cohesion Polymorphism Pure Fabrication Indirection Don't Talk to Strangers Protected Variations
Demeter es hija de Low Coupling

Ya sabemos que el acoplamiento alto es enemigo del cambio. Don't Talk to Strangers ataca una forma específica y traicionera de acoplamiento: la que se cuela aunque todas las clases estén bien encapsuladas.

¿Por qué "Demeter"?

El nombre viene del Proyecto Demeter (1987), donde se formuló la regla. La consigna popular la resume mejor: no hables con extraños — un objeto solo debería enviar mensajes a sus conocidos cercanos.

El problema

Bien encapsulado… y sin embargo acoplado

PlainSaleContent conoce solo una Sale. Pero mirá hasta dónde llega navegando:

PlainSaleContent.cs · v1
public class PlainSaleContent : ISaleContent
{
    private Sale sale;

    public PlainSaleContent(Sale sale)
    {
        this.sale = sale;
    }

    public string GetTextToPrint()
    {
        string result = $"Fecha: {this.sale.DateTime}\n";
        foreach (SalesLineItem item in this.sale)
        {
            result = result + $"{item.Quantity} de '{item.Product.Description}' " +
                $"a ${item.Product.Price}\n";
        }
        result = result + $"Total: ${this.sale.Total}";
        return result;
    }
}
La encapsulación está intacta

Sale guarda sus SalesLineItem en una lista privada, que solo se modifica vía AddLineItem / RemoveLineItem. Nadie rompió ninguna regla de visibilidad.

Y sin embargo…

Vía IEnumerable, PlainSaleContent llega a los SalesLineItem, y desde ahí a ProductSpecification. Para esta clase, esos son objetos indirectos: extraños con los que está conversando.

El problema

El "tren de llamadas" y su costo

La forma más visible de la violación es la cadena de puntos. Cada vagón agregado es una estructura interna ajena de la que ahora dependés:

item──▶ .Product──▶ .Description

El objeto que escribe esto conoce: ① que el item tiene un producto, ② que el producto tiene una descripción, ③ los tipos de ambos. Tres conocimientos que no le corresponden.

Fragilidad

Si mañana SalesLineItem deja de exponer Product, o el producto reestructura sus datos, se rompe una clase que ni siquiera es parte de esa estructura.

Cambios en cascada

Un cambio interno se propaga a todos los que navegaron la estructura. El costo de modificar deja de ser local: eso es exactamente acoplamiento alto.

Difícil de testear

Para probar la clase necesitás armar toda la cadena de objetos reales o mocks anidados: el test te está gritando que hay demasiado conocimiento estructural.

El problema en una pregunta

¿Cómo evitar conocer la estructura de objetos indirectos? Esa es la pregunta que la guía responde.

La regla

¿Con quién SÍ puede hablar un objeto?

Dentro de un método, un objeto solo debería enviar mensajes o acceder a atributos de sus objetos directos. Son exactamente estos cinco:

1 Él mismo

El propio objeto: this en C#. Hablarte a vos mismo siempre está permitido.

2 Sus argumentos

Los objetos recibidos como parámetros del método. Te los presentaron formalmente: son conocidos.

3 Sus atributos

Los objetos referenciados por variables de instancia propias.

4 Sus colecciones

Los objetos contenidos en una colección que es atributo del propio objeto.

5 Sus criaturas

Los objetos creados dentro del método (con new). Si lo creaste, lo conocés.

Todos los demás

Son extraños. Típicamente: lo que obtenés como resultado de pedirle algo a un conocido. A esos, ni mensajes ni atributos.

La solución

Delegar: que el directo hable con el indirecto

La solución de Larman: asignar la responsabilidad de colaborar con los objetos indirectos a los objetos directos. Yo le hablo a mi conocido; mi conocido le habla a los suyos.

Hablando con extraños
PlainSaleContent→ Sale→ SalesLineItem→ ProductSpecification

Un solo objeto conoce toda la cadena.

Delegando
PlainSaleContent→Sale
Sale→SalesLineItem
SalesLineItem→ProductSpecification

Cada objeto habla solo con el siguiente eslabón: conocimiento local, acoplamiento bajo.

¿Y cómo recorremos una estructura entera sin violar la regla?

Para eso existe un patrón del catálogo de la Gang of Four (Gamma, Helm, Johnson y Vlissides — Design Patterns): el patrón Visitor. Vamos a verlo en acción sobre nuestra venta.

Patrón Visitor

Visitor: la idea en dos mensajes

Accept(visitor)

Cada objeto de la estructura tiene un método Accept que recibe un Visitor. Su trabajo es mínimo: abrirle la puerta y presentarse.

Visit(objeto)

El visitante tiene un método Visit por cada tipo de la estructura. Ahí vive la lógica de qué hacer con cada uno.

el corazón del patrón
// en cada clase de la estructura:
public void Accept(Visitor visitor)
{
    visitor.Visit(this);
}
El truco: doble despacho

El objeto se pasa a sí mismo (this) como argumento. Así, cuando el visitante ejecuta su Visit, el objeto que recibe llegó como parámetro… es decir, llegó como conocido. Demeter feliz.

Patrón Visitor

Paso 1: Accept en toda la estructura

Las tres clases de la venta reciben el mismo método. Solo se muestra lo nuevo; los … representan el código que ya existía.

Sale.cs
public class Sale : IEnumerable
{
    …
    public void Accept(Visitor visitor)
    {
        visitor.Visit(this);
    }
}
SalesLineItem.cs
public class SalesLineItem
{
    …
    public void Accept(Visitor visitor)
    {
        visitor.Visit(this);

public class ProductSpecification
{
    …
    public void Accept(Visitor visitor)
    {
        visitor.Visit(this);
    }
}
Observación

El método es idéntico en las tres clases, pero no se puede heredar de una superclase común: en cada clase, this tiene un tipo distinto, y eso es justamente lo que hace que se invoque el Visit correcto (sobrecarga por tipo).

Patrón Visitor

Paso 2: la clase abstracta Visitor

Visitor.cs
public abstract class Visitor
{
    public string Content
    {
        get
        {
            return this.ContentBuilder.ToString();
        }
    }

    protected StringBuilder ContentBuilder { get; }
        = new StringBuilder();

    public abstract void Visit(Sale sale);

    public abstract void Visit(SalesLineItem lineItem);

    public abstract void Visit(ProductSpecification product);
}
  • Un Visit abstracto por cada tipo de la estructura: Sale, SalesLineItem y ProductSpecification.
  • Content expone el texto construido durante la recorrida.
  • ContentBuilder es un StringBuilder protegido: las subclases concretas lo usan para ir acumulando texto de forma eficiente.
Lo que viene

Dos visitantes concretos van a heredar de acá: uno que arma texto plano y otro que arma Markdown. Misma estructura recorrida, dos resultados distintos.

Patrón Visitor

Paso 3: un visitante concreto

PlainContentVisitor.cs
public class PlainContentVisitor : Visitor
{
    public override void Visit(Sale sale)
    {
        this.ContentBuilder.Clear();
        this.ContentBuilder.Append($"Fecha: {sale.DateTime}\n");
        foreach (SalesLineItem item in sale)
        {
            item.Accept(this);
        }

        this.ContentBuilder.Append($"Total: ${sale.Total}");
    }

    public override void Visit(SalesLineItem item)
    {
        this.ContentBuilder.Append($"{item.Quantity} de ");
        item.Product.Accept(this);
    }

    public override void Visit(ProductSpecification product)
    {
        this.ContentBuilder.Append($"'{product.Description}' a ${product.Price}\n");
    }
}
¿Por qué esto NO viola Demeter?

Cada Visit recibe el objeto con el que interactúa como argumento: es un conocido, no un extraño. El visitante nunca navega una cadena que no le pertenece — la estructura misma le va entregando cada objeto.

La danza completa

Visit(Sale) recorre los items y a cada uno le dice Accept(this) → el item responde Visit(item) → que delega en el producto con Accept(this) → que responde Visit(product). Ping-pong entre estructura y visitante.

El premio

Nuevos formatos sin tocar la estructura

MarkdownContentVisitor hereda de Visitor y produce una tabla Markdown. Sale, SalesLineItem y ProductSpecification no cambiaron ni una línea.

MarkdownContentVisitor.cs (fragmento)
public class MarkdownContentVisitor : Visitor
{
    public override void Visit(Sale sale)
    {
        this.ContentBuilder.Clear();
        this.ContentBuilder.Append($"Fecha: {sale.DateTime}\n" +
            "Qty|Prod|Price|\n" +
            "|---|:---------:|-----:|\n");

        foreach (SalesLineItem item in sale)
        {
            item.Accept(this);
        }

        this.ContentBuilder.Append($"**Total: ${sale.Total}**");
    }
    …
}
PlainSaleContent.cs · v2
public string GetTextToPrint()
{
    Visitor visitor = new PlainContentVisitor();
    this.sale.Accept(visitor);
    return visitor.Content;
}
Antes vs después

v1: GetTextToPrint navegaba item.Product.Description — conocía 3 niveles de estructura.
v2: crea un visitante, le pide a la venta que lo acepte, y lee el resultado. Conoce un solo nivel.

Extensible por diseño

¿Mañana querés HTML? ¿JSON? ¿Un ticket térmico? Una subclase nueva de Visitor y listo. La estructura de la venta queda cerrada a modificación, abierta a extensión — ¿les suena de OCP?

Cierre

Lo que te llevás de hoy

  • La encapsulación no alcanza: navegar objetos indirectos también acopla.
  • Los 5 conocidos: this, argumentos, atributos, elementos de colecciones propias y objetos creados en el método.
  • La solución es delegar: cada objeto habla solo con el eslabón siguiente.
  • Visitor recorre estructuras sin violar la ley: cada Visit recibe su objeto como argumento.
  • Bonus: la estructura queda abierta a extensión (nuevos formatos) sin modificación.
