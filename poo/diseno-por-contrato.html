Punto de partida

¿Qué significa que un programa sea correcto?

La corrección es uno de los factores de calidad de un producto de software. Pero es un concepto relativo.

Especificación

La especificación describe qué debe hacer el software, no cómo lo hace. Sin especificación, no hay forma de juzgar si algo es correcto o incorrecto.

Una pieza de software es correcta cuando satisface su especificación.
Encapsulamiento y tipos

Gracias a la encapsulación, enviamos mensajes a objetos para acceder o cambiar su estado. Los mensajes posibles están especificados en el tipo del objeto.

¿Alcanza con que la firma del método coincida con la operación del tipo para decir que es correcto?

No alcanza. La firma no dice: qué hace la operación, cómo cambia el estado, ni en qué estado debe estar el objeto para poder enviar el mensaje. Necesitamos fórmulas de corrección.
Fórmula central

La Fórmula de Corrección

Mecanismo formal para especificar qué hace una operación.

{ P }   A   { Q }
"Cualquier ejecución de A que comience en un estado donde P se cumple,
termina en un estado donde Q se cumple."
Precondición P
Afirmación que debe ser verdadera antes de ejecutar A.

Responsabilidad del que llama.
Operación A
El método o acción que se ejecuta. Si P se cumple al entrar, debe garantizar Q al salir.
Postcondición Q
Afirmación que debe ser verdadera después de ejecutar A.

Responsabilidad del que implementa.
Una afirmación es un predicado que puede ser verdadero o falso, y describe propiedades semánticas del tipo.
Precondición

La Precondición

Precondición
Una precondición es una afirmación asociada a una operación que debe ser satisfecha por la clase de los objetos que usan esa operación.
Para el cliente (que llama)

La precondición es una obligación: antes de invocar la operación, debe asegurarse de que la precondición sea verdadera.

Para el servidor (que implementa)

La precondición es un derecho: puede asumir que la precondición se cumple al inicio de su ejecución.

Ejemplo conceptual en C#
// Precondición: index >= 0 && index < Count
// El CLIENTE debe garantizarlo antes de llamar
public T Get(int index)
{
    // El método PUEDE ASUMIR que index es válido
    return this.items[index];
}

// El cliente es responsable:
if (i >= 0 && i < myList.Count)
{
    var item = myList.Get(i);   // cumple la precondición
}
Postcondición

La Postcondición

Postcondición
Una postcondición es una afirmación asociada a una operación que debe ser satisfecha por la clase de los objetos que implementan esa operación.
Para el cliente (que llama)

La postcondición es un derecho: puede confiar en que, si cumplió la precondición, el resultado satisface la postcondición.

Para el servidor (que implementa)

La postcondición es una obligación: debe garantizar que el estado resultante satisfaga la postcondición.

Ejemplo conceptual en C#
// Pre:  amount > 0 && amount <= Balance
// Post: Balance == Balance_anterior - amount
//       && el retiro fue registrado
public void Withdraw(decimal amount)
{
    // El implementador DEBE garantizar
    // que Balance se reduce en amount
    this.balance -= amount;
    this.RegisterTransaction(-amount);
    // Al terminar, Post debe ser verdadera
}
El contrato

Pre + Post = Contrato

La precondición y la postcondición forman un contrato entre el objeto que llama y el que implementa. Como un contrato legal, define derechos y obligaciones de cada parte.

Si el cliente viola la precondición, el servidor no tiene ninguna obligación. Si el servidor viola la postcondición (cuando la pre se cumplía), el programa es incorrecto.
Fortaleza

Fortaleza y debilidad de afirmaciones

Una afirmación puede ser más fuerte (más restrictiva, más difícil de satisfacer) o más débil que otra.

Fortaleza / Debilidad
Se dice que P₁ es más fuerte que P₂ (y P₂ más débil que P₁) si:

   P₁ ⟹ P₂  y  P₁ ≠ P₂

Es decir: P₁ ⟹ P₂ es equivalente a ¬P₁ ∨ P₂
Tabla de verdad de P₁ ⟹ P₂
P₁ P₂ ¬P₁ ¬P₁ ∨ P₂  (P₁⟹P₂)
VVFV
VFFF ← única fila falsa
FVVV
FFVV
Ejemplos intuitivos
  • F
    x > 0 es más fuerte que x >= 0
    (x>0 implica x≥0, pero no al revés)
  • D
    x >= 0 es más débil que x > 0
    (hay más estados que la satisfacen)
  • i
    true es la afirmación más débil posible. Cualquier estado la satisface.
Invariante

La Invariante de Tipo / Clase

Además de pre y postcondiciones para operaciones, necesitamos expresar propiedades globales que valen para todo objeto del tipo, durante toda su existencia.

Invariante de tipo / clase
Una invariante es una afirmación asociada a un tipo que debe ser satisfecha por los objetos de ese tipo durante toda su existencia.
Ejemplo: clase BankAccount
// Invariante: Balance >= 0
// Vale antes y después de cada operación pública

public class BankAccount
{
    private decimal balance;

    // Inv: balance >= 0
    public BankAccount(decimal initial)
    {
        // Precondición: initial >= 0
        this.balance = initial;
        // Post: balance == initial (inv se cumple)
    }

    // Pre: amount > 0 && amount <= Balance
    // Post: Balance == old(Balance) - amount
    //       && Balance >= 0  ← invariante se mantiene
    public void Withdraw(decimal amount)
    {
        this.balance -= amount;
    }
}
Invariante en Subtipos
La invariante de un subtipo es la conjunción (∧) de las invariantes de sus supertipos.

Es decir: el subtipo hereda todas las invariantes del supertipo y puede agregar las propias.
Una invariante puede traducirse como precondición y postcondición en todas las operaciones: debe ser verdadera al entrar y al salir de cada método público.
Especificación completa

Especificación = Pre + Post + Invariante

Con los tres elementos podemos definir formalmente la especificación de un tipo y la corrección de su implementación.

Especificación de un tipo
La especificación de un tipo es el conjunto de precondiciones, postcondiciones e invariantes asociadas a ese tipo.
Corrección
Una clase que implementa uno o más tipos es correcta cuando satisface la especificación de todos los tipos que implementa.

Un programa es correcto cuando todas sus clases son correctas.
La especificación de un programa OO

Como en un programa OO solo hay objetos enviándose mensajes, y los mensajes posibles están definidos en sus tipos, la especificación del programa es la suma de las especificaciones de todos sus tipos.

Especificación Programa
 = ∑ Especificaciones de tipos
 = ∑ (Pre + Post + Inv) por tipo
Programa correcto ⟺ todas sus clases son correctas ⟺ todas satisfacen sus especificaciones.
Design by Contract + LSP

El contrato y el Principio de Sustitución

El Diseño por Contrato está directamente relacionado con el Principio de Sustitución de Liskov (LSP): un subtipo S puede sustituir a su supertipo T sin romper el programa.

Precondiciones en Subtipos
Una operación de un subtipo debe tener precondiciones iguales o más débiles que las de su supertipo.

Pre_S ⟹ Pre_T  (Pre_T es más fuerte o igual)
Postcondiciones en Subtipos
Una operación de un subtipo debe tener postcondiciones iguales o más fuertes que las de su supertipo.

Post_T ⟹ Post_S  (Post_S es más fuerte o igual)
Idea central: un cliente que antes cumplía Pre_T también cumplirá Pre_S (es más fácil). Un servidor que cumple Post_S también cumple Post_T (da más de lo prometido).
Ejemplo integrador

Pre, Post e Invariante en código

Ejemplo completo con comentarios de especificación: cuenta bancaria con tipo base y subtipo.

BankAccount.cs — especificación en comentarios
// INVARIANTE: Balance >= 0
public class BankAccount
{
    public decimal Balance { get; private set; }

    // Pre:  initialBalance >= 0
    // Post: Balance == initialBalance
    public BankAccount(decimal initialBalance)
    {
        if (initialBalance < 0)
            throw new ArgumentException("Balance inicial negativo");
        this.Balance = initialBalance;
    }

    // Pre:  amount > 0  &&  amount <= Balance
    // Post: Balance == old(Balance) - amount  &&  Balance >= 0
    public virtual void Withdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance)
            throw new ArgumentException("Monto inválido");
        this.Balance -= amount;
    }
}

// Subtipo con precondición más débil (permite descubierto)
// INVARIANTE: Balance >= -OverdraftLimit  (más débil, ∧ con heredada)
public class OverdraftAccount : BankAccount
{
    private decimal overdraftLimit;

    // Pre:  amount > 0  &&  amount <= Balance + OverdraftLimit
    //       ← MÁS DÉBIL que la de BankAccount
    // Post: Balance == old(Balance) - amount  &&  Balance >= -OverdraftLimit
    //       ← MÁS FUERTE: garantiza también el límite de descubierto
    public override void Withdraw(decimal amount)
    {
        if (amount <= 0 || amount > Balance + overdraftLimit)
            throw new ArgumentException("Excede límite de descubierto");
        this.Balance -= amount;
    }
}
OverdraftAccount puede sustituir a BankAccount según LSP: su precondición es más débil (acepta más casos) y su postcondición es más fuerte (garantiza más).
Violación

Violación de afirmación

Violación de afirmación
La violación de una afirmación es la situación a la que se llega en un programa incorrecto cuando no se cumple una precondición, postcondición o invariante.

Es producto de un error permanente en el código del programa.
Violaciones en C#
var account = new BankAccount(100);

// Violación de PRECONDICIÓN:
// El cliente no cumplió Pre (amount > 0)
account.Withdraw(-50);   // ArgumentException

// Violación de INVARIANTE:
// Aunque el cliente cumplió Pre,
// si la implementación rompe la invariante
// → bug en el servidor

// Violación de POSTCONDICIÓN:
// Si Withdraw no resta el monto correctamente
// → Post no se cumple → bug del implementador
¿Quién tiene el bug?
  • P
    Precondición violada: bug en el cliente (llamó sin cumplir su obligación).
  • Q
    Postcondición violada (cuando pre se cumplía): bug en el servidor (no garantizó su obligación).
  • I
    Invariante violada: bug en el servidor (alguna operación dejó el objeto en estado inválido).
Las violaciones son señal de un programa incorrecto. Son distintas de las excepciones de usuario: no se pueden resolver sin corregir el código.
Cierre

Resumen: Diseño por Contrato

Precondición Postcondición Invariante
  • P
    Pre: obligación del cliente, derecho del servidor.
  • Q
    Post: derecho del cliente, obligación del servidor.
  • I
    Inv: verdadera para todo objeto del tipo durante toda su existencia.
  • F
    {P} A {Q}: si P al entrar → Q al salir.
Con subtipos (LSP)
  • ↓
    Pre del subtipo ≤ Pre del supertipo (más débil o igual)
  • ↑
    Post del subtipo ≥ Post del supertipo (más fuerte o igual)
  • ∧
    Inv subtipo = Inv supertipo ∧ nuevas invariantes
Esto garantiza la sustituibilidad: donde hay un T, puede haber un S sin romper el programa.

Sin especificación no hay corrección. El contrato es la especificación: Pre + Post + Invariante.
