Terminología · IEEE 610.12

¿Por qué tanta terminología?

Hablar con precisión sobre errores nos ayuda a decidir qué se puede manejar y qué es una falla irrecuperable.

Término popular Término preciso Definición
Error humano Equivocación (mistake) Acción humana que produce un resultado incorrecto.
Bug / defecto Falta (fault) Definición incorrecta en un paso, proceso o dato. Manifestación de una equivocación.
Crash / fallo Falla (failure) Comportamiento o resultado incorrecto. Resultado de una falta.
Diferencia calculada Error Cantidad por la que el resultado es incorrecto respecto del valor teórico.
Cadena causal: equivocación → falta en el código → error en los datos → falla del sistema.
Si la equivocación es del usuario, podemos manejarla (no hay falla). Si es del programador, hay falla y el programa no puede continuar.
Decisión de diseño

¿Es manejable o es una falla?

La decisión la toma el objeto que recibe el resultado de enviar un mensaje, no la clase que lanza la excepción.

Excepción manejable

La equivocación es del usuario o de otro sistema externo. Se puede subsanar sin generar una nueva versión del programa.

FileNotFoundException
→ pedir nuevamente la ruta al usuario

Ocurre una falta, pero no una falla.

Excepción no manejable (falla)

La equivocación es del programador. El programa hace algo que no está de acuerdo con su especificación. No se puede resolver sin una nueva versión.

NullReferenceException
→ la variable debería tener un objeto (especificación)

Ocurre una falla: el programa no puede continuar.

Ejemplo: FileNotFoundException puede ser manejable si la ruta la ingresó el usuario, pero podría ser una falla si el archivo es un recurso interno del sistema que debería existir siempre.
Estructura de control

La estructura try … catch … finally

Controla el flujo de ejecución cuando puede ocurrir una excepción.

try { }

Sentencias en las que puede ocurrir la excepción que me interesa manejar.

catch (Ex e) { }

Se ejecuta solo si ocurre una excepción en el bloque try. Se puede concatenar varios catch.

finally { }

Se ejecuta siempre, haya o no excepción. Típicamente para liberar recursos.

Si no hay ningún try…catch que capture la excepción, se busca en el método que invocó al actual, y así hasta el programa principal. Si nadie la captura, el programa termina.
Código de ejemplo

Ejemplo: abrir un archivo

Escritura en un archivo: la apertura puede lanzar UnauthorizedAccessException. El finally garantiza que el archivo se cierre siempre.

FileExample.cs
FileStream file = null;
FileInfo fileinfo = new FileInfo(@"c:\file.txt");

try
{
    // Puede lanzar: UnauthorizedAccessException, IOException, etc.
    file = fileinfo.OpenWrite();
    file.WriteByte(0xF);
}
catch (UnauthorizedAccessException e)
{
    // Solo captura esta excepción específica
    Debug.WriteLine("Acceso no autorizado");
    throw e;     // ← re-lanza la excepción para el llamador
}
finally
{
    // Se ejecuta SIEMPRE: libera el recurso
    if (file != null)
    {
        file.Close();
    }
}
throw e

Relanza la excepción al llamador.

throw

Sin argumento: relanza la excepción actual sin perder el stack trace.

catch { }

Sin tipo: captura cualquier excepción.

Jerarquía

La clase System.Exception

En C# todas las excepciones heredan de System.Exception. Las clases propias también deben hacerlo.

NullReferenceException → típicamente falla (bug del programador).
FileNotFoundException → puede ser manejable (error del usuario).
Excepción propia

Crear LibraryException

No siempre es necesario crear una clase de excepción nueva, pero cuando las existentes no expresan bien el dominio del problema, es buena práctica hacerlo.

LibraryException.cs
[Serializable]              // atributo obligatorio
internal class LibraryException : Exception
{
    // Constructor sin parámetros (obligatorio)
    public LibraryException() { }

    // Constructor con mensaje (el más usado)
    public LibraryException(string message)
        : base(message) { }

    // Constructor con mensaje + excepción encadenada
    public LibraryException(string message, Exception innerException)
        : base(message, innerException) { }

    // Constructor para serialización (obligatorio)
    protected LibraryException(SerializationInfo info, StreamingContext context)
        : base(info, context) { }
}
Cuatro constructores obligatorios: sin parámetros · con mensaje · con mensaje + inner · serialización.
[Serializable] es un atributo que permite que la excepción se serialice (ej.: en llamadas remotas o logs binarios).
Caso de estudio · Library

Lanzar la excepción en el constructor

La especificación dice que no pueden existir canciones del género "marcha". Si alguien intenta crearla, es una violación de la especificación → se lanza LibraryException.

Song.cs
public class Song : Media
{
    private string artist;

    public Song(string name, string genre, string artist)
        : base(name, genre)
    {
        // Validación de la regla de negocio en el constructor
        if (string.Equals(genre, "marcha", StringComparison.OrdinalIgnoreCase))
        {
            // La excepción es un objeto: se usa new
            throw new LibraryException(
                $"No me gusta {name} porque no me gusta la marcha");
        }
        this.artist = artist;
    }
}
throw new LibraryException(...) crea el objeto excepción y lo lanza en un solo paso.
StringComparison.OrdinalIgnoreCase hace la comparación insensible a mayúsculas: "Marcha", "MARCHA", "marcha" → todas bloqueadas.
El constructor es el lugar ideal para validar invariantes: si la construcción del objeto falla, el objeto nunca existe en un estado inválido.
Capturar la excepción

El programa principal captura LibraryException

El programa intenta agregar canciones, incluyendo una de marcha. El catch captura la excepción y el finally imprime la biblioteca en cualquier caso.

Program.cs
public class Program
{
    private static Library library = new Library();

    public static void Main(string[] args)
    {
        try
        {
            AddMovies();         // agrega películas
            AddSongs();          // agrega canciones (no-marcha)
            AddMarchaSongs();    // ← lanza LibraryException
        }
        catch (LibraryException e)
        {
            // Solo captura LibraryException (y subclases)
            Console.WriteLine(e.Message);
            // Las canciones agregadas ANTES de la excepción permanecen
        }
        finally
        {
            // Se ejecuta siempre: imprime el estado final de la biblioteca
            PrintMediaItems("La biblioteca contiene:", library);
        }
    }
}
Efecto importante: cuando AddMarchaSongs() lanza la excepción, las sentencias siguientes dentro de try no se ejecutan. El control pasa directamente al catch.
Flujo de ejecución

¿Qué pasa exactamente al lanzar?

Trazamos el flujo de ejecución desde que Song lanza la excepción hasta que Program la captura.

Flujo de stack unwinding: la excepción sube por el call stack hasta encontrar un catch adecuado

Múltiples catch

Concatenar bloques catch

Es posible manejar excepciones de distintas clases en el mismo bloque try, con un catch por cada tipo. Se evalúan en orden: de más específica a más general.

MultiCatch.cs
try
{
    string ruta = Console.ReadLine();         // el usuario ingresa una ruta
    File.WriteAllText(ruta, "contenido");
}
catch (UnauthorizedAccessException e)
{
    // Sin permiso de escritura
    Console.WriteLine($"Sin permiso: {e.Message}");
}
catch (DirectoryNotFoundException e)
{
    // El directorio no existe
    Console.WriteLine($"Directorio no encontrado: {e.Message}");
}
catch (IOException e)
{
    // Cualquier otro error de I/O (más general)
    Console.WriteLine($"Error de E/S: {e.Message}");
}
catch (Exception e)
{
    // Catch-all: captura cualquier excepción no capturada antes
    Console.WriteLine($"Error inesperado: {e.Message}");
    throw;    // re-lanzar sin modificar el stack trace
}
finally
{
    Console.WriteLine("Operación finalizada.");
}
Orden importa: siempre de más específica a más general. Si pusieras catch (Exception) primero, los catch siguientes nunca se ejecutarían (error de compilación en C#).
Buenas prácticas

Resumen y buenas prácticas

Sí hacer
  • ✓
    Lanzar excepciones específicas del dominio con mensajes descriptivos.
  • ✓
    Usar finally para liberar recursos (archivos, conexiones, memoria).
  • ✓
    Validar invariantes en el constructor: objeto inválido = excepción.
  • ✓
    Capturar solo las excepciones que puedas manejar correctamente.
  • ✓
    Usar throw; (sin argumento) para no perder el stack trace original.
No hacer
  • ✗
    No usar excepciones para flujo de control normal (reemplazar if).
  • ✗
    No silenciar excepciones con catch { } vacío.
  • ✗
    No capturar Exception base si no vas a relanzarla.
  • ✗
    No lanzar NullReferenceException manualmente: usar ArgumentNullException.
  • ✗
    No usar throw e; si quieres preservar el stack trace: usa throw;.

Las excepciones expresan violaciones de la especificación. Úsalas para comunicar situaciones excepcionales, no para flujo normal.
