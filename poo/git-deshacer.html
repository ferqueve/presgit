Cómo revertir un commit en Git

Presentación práctica con explicaciones claras, casos reales y ejercicios.

git revert git reset HEAD~1 push --force

Usa las flechas del teclado o los botones inferiores para avanzar.

¿Qué significa revertir un commit?

Significa deshacer cambios que quedaron registrados en el historial de Git.

Pero hay dos caminos muy distintos:

  • git revert: deshace sin borrar historial.
  • git reset: mueve la rama hacia atrás y puede borrar historial.
Idea clave: no siempre “deshacer” significa lo mismo. Primero debes decidir si quieres conservar el historial o reescribirlo.

Pregunta guía

Antes de ejecutar un comando, pregúntate:

  1. ¿Ya hice push?
  2. ¿Trabajo solo o en equipo?
  3. ¿Quiero conservar los cambios para editarlos?
  4. ¿Quiero dejar rastro de lo que pasó?

Opción 1: git revert

Crea un nuevo commit que deshace los cambios de otro commit anterior.

git revert <commit_id>
Ventajas
  • No borra historial.
  • Es seguro en repositorios compartidos.
  • Permite auditar qué se hizo y qué se deshizo.
Cuándo usarlo
  • Cuando ya hiciste push.
  • Cuando otras personas ya pueden haber descargado esos commits.
  • Cuando quieres una solución segura y profesional.
Caso real: subiste un cambio que rompió el sistema de login en producción. Como el commit ya está en el remoto y otros compañeros trabajan sobre esa rama, lo correcto suele ser git revert.

Opción 2: git reset

Mueve el puntero de la rama hacia atrás. Dependiendo del modo, conserva o elimina cambios.

Comando Qué hace Riesgo
git reset --soft HEAD~1 Deshace el commit, pero deja los cambios en staging. Bajo
git reset --mixed HEAD~1 Deshace el commit y deja cambios sin stage. Medio
git reset --hard HEAD~1 Deshace el commit y borra cambios locales. Alto
Atención: --hard puede borrar trabajo no guardado. No lo uses a la ligera.

1. --soft

Ideal cuando te equivocaste en el commit, pero quieres rearmarlo.

git reset --soft HEAD~1

2. --mixed

Deshace el commit y deja los archivos modificados para que decidas qué volver a agregar.

git reset --mixed HEAD~1

3. --hard

Vuelve atrás y elimina todo lo no confirmado desde ese punto.

git reset --hard HEAD~1
Regla simple

Si quieres seguridad, piensa primero en revert. Si quieres reescribir historial local y sabes lo que haces, usa reset.

¿Qué pasa si ya hice push?

Camino recomendado
git revert <commit_id>

No rompe el historial compartido.

Camino riesgoso
git reset --hard <commit_id> git push --force

Puede desordenar el trabajo de otros si ya basaron cambios sobre esos commits.

En equipos, forzar push suele requerir mucho cuidado, coordinación y permisos.

Casos reales de trabajo

Caso 1: Hotfix equivocado en producción

Un desarrollador sube un parche rápido y rompe el formulario de pagos.

Conviene: git revert

Por qué: el cambio ya está publicado y no conviene reescribir historial compartido.

Caso 2: Commit local con mensaje malo o archivos de más

Hiciste commit, pero olvidaste excluir un archivo temporal y el mensaje quedó mal.

Conviene: git reset --soft HEAD~1

Por qué: puedes volver a commitear correctamente sin perder cambios.

Caso 3: Estabas probando y no quieres guardar nada

Hiciste cambios locales experimentales, no sirven y quieres volver al último estado limpio.

Conviene: git reset --hard HEAD~1 o incluso descartar cambios según el caso.

Caso 4: Ya subiste, pero estás solo en tu rama personal

Puede ser válido usar push --force, pero sigue siendo recomendable saber exactamente qué impacto tendrá.

Pasos sugeridos antes de revertir

Paso 1

Revisa el historial con git log --oneline.

Paso 2

Identifica el commit correcto.

Paso 3

Define si el repo es compartido o solo tuyo.

Paso 4

Decide si conservarás los cambios o los eliminarás.

Paso 5

Ejecuta el comando adecuado.

Paso 6

Verifica con git status y prueba el proyecto.

Soluciones explicadas

Ejercicio 1

Respuesta correcta: git revert <commit_id>

Porque el commit ya fue compartido. Lo mejor es deshacerlo sin borrar el historial de la rama.

Ejercicio 2

Respuesta correcta: git reset --soft HEAD~1

Porque quieres deshacer el último commit, pero conservar los cambios listos para volver a commitearlos correctamente.

Consejo profesional: antes de un reset importante, puedes crear una rama de respaldo por seguridad.
git branch respaldo-antes-de-reset

Resumen final

Usa revert para deshacer con seguridad. Usa reset cuando necesites reescribir historial y entiendas el impacto.

# Seguro para trabajo en equipo git revert <commit_id> # Rehacer el último commit sin perder cambios git reset --soft HEAD~1 # Volver atrás descartando todo git reset --hard HEAD~1
