Git: cómo revertir un commit o un push equivocado

Presentación pensada para clase: explicación paso a paso, casos reales y ejercicios.

git add git commit git push git revert git reset

Puedes avanzar con los botones inferiores o con las flechas del teclado.

El proceso completo de trabajo en Git

1. Modificas archivos
Ejemplo: cambias un archivo HTML, CSS o JS.
2. Preparas los cambios
Usas git add para enviarlos al área de staging.
3. Confirmas los cambios
Usas git commit para guardarlos en el historial local.
4. Publicas en remoto
Usas git push para subirlos a GitHub, GitLab o similar.

Idea clave para clase

Un error puede ocurrir en cualquiera de estas etapas:

  • Antes del commit
  • Después del commit pero antes del push
  • Después del push
La forma correcta de deshacer algo depende de en qué etapa te equivocaste.

¿Qué significa “revertir”?

En Git, deshacer un error no siempre significa lo mismo. A veces quieres:

Conservar historial

Se usa normalmente git revert.

Rehacer el último commit

Se usa con frecuencia git reset --soft.

Borrar todo

Se usa con mucho cuidado git reset --hard.

Antes de elegir un comando, conviene responder tres preguntas: ¿ya hice push?, ¿trabajo solo o en equipo?, ¿quiero conservar los cambios o eliminarlos?

Escenario central: commit y push equivocado

Supongamos este caso típico para explicarlo en clase:

1) Modificaste login.js 2) Hiciste git add . 3) Hiciste git commit -m "Arreglo login" 4) Hiciste git push 5) Descubres que rompiste el acceso de usuarios
En este caso, el error ya está en el remoto. Si otros pueden ver o descargar ese commit, la opción más segura normalmente es git revert.

Paso a paso para revertir un commit ya pusheado

Paso 1: ver el historial
git log --oneline

Esto te muestra los commits recientes en formato resumido.

Paso 2: identificar el commit incorrecto
a1b2c3d Arreglo login x8y9z7w Cambio de estilos r4t5u6v Actualización footer

Aquí, por ejemplo, a1b2c3d es el commit problemático.

Paso 3: revertirlo
git revert a1b2c3d

Git creará un nuevo commit que deshace ese cambio.

Paso 4: subir el revert
git push

Ahora el remoto queda con el historial intacto, pero con el error deshecho.

¿Qué ocurre internamente con git revert?

No borra el commit original. Lo que hace es agregar otro commit que compensa sus cambios.

Antes
Commit A Commit B (error) Commit C
Después del revert
Commit A Commit B (error) Commit C Commit D (revert de B)
Esto es muy útil en equipos porque el historial queda claro: se ve qué se hizo y cómo se corrigió.

¿Cuándo aparece git reset?

git reset se usa más cuando quieres mover la rama hacia atrás. Es común en trabajo local, antes de compartir.

Comando Resultado Uso típico
git reset --soft HEAD~1 Quita el commit, pero deja cambios en staging Rehacer el commit
git reset --mixed HEAD~1 Quita el commit y deja cambios sin stage Revisar antes de volver a agregar
git reset --hard HEAD~1 Quita commit y borra cambios locales Descartar por completo
Si ya hiciste push y trabajas con otras personas, usar reset más force push puede ser problemático.

Caso 1

Hiciste commit, pero no push.

Quieres corregir el mensaje o agregar un archivo faltante.

git reset --soft HEAD~1

Te deja todo listo para commitear nuevamente.

Caso 2

Hiciste commit y push equivocado.

El cambio rompió producción o una funcionalidad importante.

git revert <commit_id> git push

Es la salida más segura en ramas compartidas.

Caso 3

Solo estás probando localmente.

Quieres borrar todo y volver al punto anterior.

git reset --hard HEAD~1

Úsalo solo si entiendes que perderás esos cambios.

Caso de trabajo en equipo

Situación

Subiste un cambio a la rama compartida y dos compañeros ya hicieron pull.

La opción más sana suele ser git revert.

Problema si fuerzas

Si haces git push --force, puedes dejar a tus compañeros con historiales distintos y conflictos difíciles de resolver.

En clase, esta es una buena regla simple: si el commit ya fue compartido, primero piensa en revert y no en reset.

¿Y si ya hice push pero estoy solo en mi rama?

En ese caso, a veces se usa reescritura de historial.

git reset --hard HEAD~1 git push --force
Ventaja

La rama queda como si ese commit nunca hubiera existido.

Riesgo

Si alguien más alcanzó a usar esa rama, puedes causar problemas.

Este método no suele ser el primero que se enseña a principiantes para trabajo colaborativo. Para clase conviene presentarlo como una opción avanzada y delicada.

Checklist antes de deshacer un commit

1

Revisa git status

2

Consulta git log --oneline

3

Confirma si ya hubo push

4

Piensa si trabajas solo o en equipo

5

Decide si quieres conservar cambios

6

Haz prueba final después del cambio

Resumen para recordar en clase:
• Si ya hiciste push y compartiste el commit: piensa primero en git revert.
• Si todavía no hiciste push: muchas veces git reset te sirve mejor.
• Si usas --hard, puedes perder trabajo.
