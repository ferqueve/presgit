El Problema
Pregunta central de Creator

Todos los objetos deben ser creados para poder enviarles mensajes que hagan uso de sus responsabilidades de hacer y de conocer. ¿Quién debería ser responsable de crear una nueva instancia de cierta clase?

Objetos necesitan ser instanciados

Antes de enviar mensajes a un objeto, este debe existir en memoria. Alguien debe ser responsable de invocar new ClaseA().

Sin guía → decisiones arbitrarias

Si no hay criterio claro, la responsabilidad de crear se asigna al azar: a Program, a clases de utilidad, o dispersada por todo el sistema.

Consecuencias del mal diseño

Alta dependencia entre clases no relacionadas, objetos creados que no pertenecen a nadie, dificultad para mantener invariantes del sistema.

Creator responde la pregunta

Provee un criterio basado en relaciones existentes entre clases para determinar quién debe crear a quién.

La Solución – Condiciones de Creator
Principio general

Asignar a la clase B la responsabilidad de crear una instancia de la clase A si una o más de las siguientes condiciones son verdaderas:

1
B agrega objetos A

B es una colección o lista que agrega instancias de A. Ej: Pedido agrega LineaPedido.

2
B contiene objetos A

B tiene a A como parte de su composición. La relación es de contenencia o composición fuerte.

3
B guarda instancias de A

B actúa como repositorio, almacén o registro que persiste instancias de A.

4
B usa de forma cercana A

B tiene una relación de uso muy estrecha con A: siempre los usa juntos, no uno sin el otro.

5
B tiene los datos para crear A

B posee toda la información necesaria para inicializar las instancias de A. B es el Expert en crear A.

Preferencia: Cuando se pueda aplicar más de una alternativa, prefieran la clase B que agrega o contiene instancias de A. En cualquier caso, B tiene una variable de instancia de A o un contenedor capaz de almacenar instancias de A.

Ejemplo – Punto de Venta (Antes)
Contexto

Del ejemplo Expert y SRP: tenemos una aplicación de punto de venta. La clase Sale agrega instancias de SalesLineItem. Originalmente, Program crea las instancias de SalesLineItem.

v1 — Program crea SalesLineItem (MAL)
Program.cs C# 
public class Program
{
    public static void Main(string[] args)
    {
        // Program crea SalesLineItem
        // → VIOLA Creator
        sale.AddLineItem(
          new SalesLineItem(1, ProductAt(0)));
        sale.AddLineItem(
          new SalesLineItem(2, ProductAt(1)));
        sale.AddLineItem(
          new SalesLineItem(3, ProductAt(2)));
    }
}
Problema

Program no agrega, contiene ni usa directamente a SalesLineItem. La clase que realmente lo hace es Sale. Esto genera acoplamiento innecesario.

Diagrama v1

Program NO tiene relación natural con SalesLineItem
Ejemplo – Punto de Venta (Después)
v2 — Sale crea SalesLineItem (BIEN)
Sale.cs C# 
public class Sale
{
  private List<SalesLineItem> lineItems
      = new List<SalesLineItem>();

  public SalesLineItem AddLineItem(
      double quantity,
      ProductSpecification product)
  {
    SalesLineItem item =
        new SalesLineItem(quantity, product);
    this.lineItems.Add(item);
    return item; // menos huérfanos
  }
}
Program.cs (refactored) C# 
// Program ya no crea SalesLineItem
sale.AddLineItem(1, ProductAt(0));
sale.AddLineItem(2, ProductAt(1));
sale.AddLineItem(3, ProductAt(2));
Diagrama v2
Sale
- lineItems: List<SalesLineItem>
+ AddLineItem(qty, prod)
+ RemoveLineItem(item)
↓ crea y agrega
SalesLineItem
- quantity: double
- product: ProductSpec
+ GetTotal()
Sale agrega SalesLineItem → es su Creator natural

El método AddLineItem también garantiza que no existan instancias de SalesLineItem huérfanas, pues las agrega inmediatamente.

Resumen – Regla Creator
Fórmula del patrón

Si B agrega, contiene, guarda, usa cercanamente A, o tiene los datos para inicializar A → entonces B es el Creator de A.

Beneficios
  • Reduce el acoplamiento entre clases no relacionadas
  • Los objetos creados son automáticamente parte de una estructura
  • Soporta la encapsulación y los invariantes del sistema
  • Hace más difícil la creación de objetos huérfanos
  • Facilita la comprensión del modelo de dominio
Anti-patrones a evitar
  • Que Program cree objetos que "pertenecen" a otra clase
  • Clases de utilidad crean objetos de dominio
  • Objetos creados sin ser asignados a ningún contenedor
  • Lógica de construcción dispersa por todo el sistema
Relación con otros patrones

Expert: Creator a veces usa Expert para decidir quién crea. El que tiene los datos (Expert) suele ser también el que debe crear (Creator).
Factory: Cuando la creación es compleja, Creator puede delegar a una Factory, pero el iniciador de la creación sigue siendo el que contiene o agrega.

Book.cs — Creator de BookCopy C# 
namespace BibliotecaApp;

public class Book
{
  public string Title { get; }
  public string Author { get; }
  public string ISBN { get; }

  private List<BookCopy> copies =
      new List<BookCopy>();

  public Book(string title,
               string author,
               string isbn)
  {
    Title = title;
    Author = author;
    ISBN = isbn;
  }

  // Creator: Book agrega BookCopy
  public BookCopy AddCopy(string copyId)
  {
    BookCopy copy = new BookCopy(copyId, this);
    copies.Add(copy);
    return copy;
  }

  public IEnumerable<BookCopy> GetCopies()
      => copies.AsReadOnly();

  public BookCopy? GetAvailableCopy()
      => copies.FirstOrDefault(c => c.IsAvailable);
}
Library.cs — Creator de Book C# 
namespace BibliotecaApp;

public class Library
{
  public string Name { get; }
  private List<Book> catalog =
      new List<Book>();
  private List<Member> members =
      new List<Member>();

  public Library(string name)
  { Name = name; }

  // Creator: Library agrega Book
  public Book AddBook(string title,
      string author, string isbn)
  {
    Book book = new Book(title,
                           author, isbn);
    catalog.Add(book);
    return book;
  }

  // Creator: Library agrega Member
  public Member RegisterMember(
      string name, string id)
  {
    Member m = new Member(name, id);
    members.Add(m);
    return m;
  }
}
BookCopy.cs C# 
namespace BibliotecaApp;

public class BookCopy
{
  public string CopyId { get; }
  public Book Book { get; }
  public bool IsAvailable { get; private set; } = true;

  // Internal: creado por Book
  internal BookCopy(string copyId, Book book)
  { CopyId = copyId; Book = book; }

  public void CheckOut()
      => IsAvailable = false;
  public void Return()
      => IsAvailable = true;
}

ShoppingCart.csC#
public class ShoppingCart
{
  private List<CartItem> items = new();
  public string CustomerId { get; }

  public ShoppingCart(string customerId)
  { CustomerId = customerId; }

  // Creator: agrega CartItem
  public CartItem AddItem(
      Product product, int qty)
  {
    var item = new CartItem(product, qty);
    items.Add(item);
    return item;
  }

  public decimal GetTotal()
    => items.Sum(i => i.GetSubtotal());
}

public class Patient
{
  public string Name { get; }
  public string PatientId { get; }
  private List<MedicalRecord> records = new();

  public Patient(string name, string id)
  { Name = name; PatientId = id; }

  // Creator: guarda y agrega registros
  public MedicalRecord AddRecord(
      DateTime date, string diagnosis)
  {
    var record = new MedicalRecord(
        date, diagnosis, this);
    records.Add(record);
    return record;
  }
}
