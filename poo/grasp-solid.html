Motivación

¿Por qué necesitamos principios?

El código sin principios envejece mal — estos son los síntomas

SIN PRINCIPIOS — SÍNTOMAS
Código rígido
Cambiar una cosa rompe diez. Miedo a modificar porque "quién sabe qué se cae".
Código frágil
Los cambios causan errores en partes no relacionadas. Bugs en cascada.
Código inmovilizado
Imposible reutilizar módulos porque vienen "pegados" a otras partes del sistema.
Clases Dios
Una clase que sabe todo y hace todo. Miles de líneas, responsabilidades mezcladas.
CON PRINCIPIOS — BENEFICIOS
Bajo acoplamiento
Los módulos son independientes. Cambiás uno sin tocar los demás.
Alta reutilización
Clases enfocadas que podés llevar a otros proyectos sin problema.
Más testeable
Clases pequeñas con una responsabilidad son fáciles de testear con xUnit.
GRASP te dice quién hace qué. SOLID te dice cómo diseñarlo bien.

GRASP · 01

Information Expert

Asignar la responsabilidad a la clase que tiene la información necesaria para cumplirla

Pregunta clave: ¿Qué clase tiene la información para resolver esto?
Esa clase es la Experta — y es la que debe tener el método.
MAL — responsabilidad fuera de lugar
// Pedido calcula su propio total
// ¡pero no tiene los precios!
public class Pedido
{
    public List<string> Items { get; set; }
    // ❌ No sabe los precios de los items
    public decimal CalcularTotal() { ... }
}
BIEN — Information Expert
public class LineaPedido
{
    public Producto Producto { get; set; }
    public int Cantidad { get; set; }

    // ✅ LineaPedido TIENE precio y cantidad
    public decimal CalcularSubtotal()
        => Producto.Precio * Cantidad;
}

public class Pedido
{
    public List<LineaPedido> Lineas { get; set; }

    // ✅ Pedido TIENE las líneas
    public decimal CalcularTotal()
        => Lineas.Sum(l => l.CalcularSubtotal());
}
¿CÓMO APLICARLO?
1
Identificá la responsabilidad
¿Qué cálculo o acción necesitás realizar? Ej: "calcular el subtotal de una línea de pedido".
2
¿Qué clase tiene esa info?
¿Quién tiene el precio? ¿Quién tiene la cantidad? Esa clase es la experta.
3
Asigná el método ahí
El método va en la clase experta. Si la info está distribuida, componer delegando.
Beneficio: Evita que objetos tengan que "pedir prestada" información de otros para hacer su trabajo, lo que aumenta el acoplamiento.
Relación con SOLID
Information Expert refuerza el principio S (Single Responsibility): la clase que tiene los datos es la responsable de operar con ellos.

GRASP · 02

Creator

¿Quién debería ser responsable de crear una instancia de clase B?

Asignale la responsabilidad de crear B a A si:

A
A contiene o agrega a B
Un Pedido contiene LineaPedidos → Pedido las crea.
B
A registra instancias de B
Una clase que lleva un historial es buena candidata a crear esas entradas.
C
A usa de cerca a B
Si A necesita B inmediatamente y lo usa intensamente, puede ser su creador.
D
A tiene los datos init de B
Si A tiene todos los datos necesarios para inicializar B, tiene sentido que lo cree.
Creator — Pedido crea LineaPedido 
public class Pedido
{
    private List<LineaPedido> _lineas = new();

    // ✅ Creator: Pedido CONTIENE LineaPedido
    // Pedido tiene los datos para crearla
    public void AgregarProducto(
        Producto producto, int cantidad)
    {
        // Pedido es el Creator de LineaPedido
        var linea = new LineaPedido(
            producto, cantidad);
        _lineas.Add(linea);
    }

    public decimal CalcularTotal()
        => _lineas.Sum(l => l.CalcularSubtotal());
}

// ❌ Esto violaría Creator:
// var linea = new LineaPedido(p, 2);
// pedido.AgregarLinea(linea); // el cliente crea
¿Por qué no siempre usar Factory? Creator es la solución más simple. Solo usás Factory cuando la creación es compleja, hay muchas variantes, o necesitás desacoplar más.

GRASP · 03

Controller

¿Quién recibe y coordina los eventos del sistema?

El Controller no hace el trabajo — delega a otros objetos. Es el coordinador, no el trabajador.
Opciones válidas de Controller
Facade Controller: representa todo el sistema. Ej: SistemaVentas
Use-Case Controller: uno por caso de uso. Ej: GestionarPedidoController
MAL — la UI hace la lógica
// ❌ El botón de la UI hace cálculos
private void btnPagar_Click(...)
{
    decimal total = 0;
    foreach (var item in carrito)
        total += item.Precio * item.Cantidad;
    if (total > saldo) throw new Exception("...");
    // toda la lógica de negocio en la UI
}
BIEN — Controller GRASP
// ✅ El Controller recibe el evento
// y delega a los objetos de dominio
public class GestionarPedidoController
{
    private readonly IPedidoServicio _servicio;

    public GestionarPedidoController(
        IPedidoServicio servicio)
    {
        _servicio = servicio;
    }

    // Recibe el evento del sistema
    public ResultadoPago ProcesarPago(
        int pedidoId, MetodoPago metodo)
    {
        // Delega — no hace el trabajo
        var pedido = _servicio.ObtenerPedido(pedidoId);
        return _servicio.ProcesarPago(pedido, metodo);
    }
}

// La UI solo llama al Controller:
// _controller.ProcesarPago(pedidoId, metodo);
En ASP.NET Core, los Controllers de MVC son justamente este patrón aplicado al contexto web.
GRASP · 04 + 05

Low Coupling & High Cohesion

Los dos principios GRASP que funcionan como brújula — guían todas las otras decisiones de diseño

Low Coupling
Una clase con bajo acoplamiento tiene pocas dependencias hacia otras clases. Si A cambia, B no se entera.
Inyección de dependencias → bajo acoplamiento
// ✅ Bajo acoplamiento: depende de interfaz
public class PedidoServicio
{
    private readonly IRepositorioPedido _repo;

    public PedidoServicio(IRepositorioPedido repo)
    {
        _repo = repo; // depende de abstracción
    }
}
// ❌ Alto acoplamiento: new concreto dentro
// _repo = new SqlRepositorioPedido(); ← mal
High Cohesion
Una clase con alta cohesión hace cosas relacionadas entre sí. Sus métodos trabajan con los mismos datos.
Alta cohesión vs. clase Dios
// ❌ Baja cohesión — clase Dios
public class GestorGeneral
{
    public void EnviarEmail() { }
    public decimal CalcularIVA() { }
    public void GenerarPDF() { }
    public void ValidarUsuario() { }
}

// ✅ Alta cohesión — una responsabilidad
public class EmailSender { 
    public void Enviar(Email email) { }
    public void EnviarMasivo(List<Email> e) { }
}
public class CalculadorImpuestos {
    public decimal CalcularIVA(decimal base) { }
}
Son complementarios: aumentar la cohesión suele reducir el acoplamiento. Si una clase hace poco, tiene menos razones para depender de otras.

GRASP · 06

Polymorphism GRASP

Cuando el comportamiento varía según el tipo, usá polimorfismo — no ifs ni switch

MAL — condicionales por tipo
// ❌ Cada nuevo tipo de pago = modificar esto
public void ProcesarPago(string tipo, decimal monto)
{
    if (tipo == "efectivo")
        PagarEfectivo(monto);
    else if (tipo == "tarjeta")
        PagarTarjeta(monto);
    else if (tipo == "transferencia")
        PagarTransferencia(monto);
    // ¿y si agregamos crypto? ← modificar acá
}
Cada nuevo tipo de pago requiere modificar la misma función. Esto viola también Open/Closed (SOLID-O).
BIEN — Polimorfismo GRASP
// ✅ Interfaz / clase abstracta
public interface IMetodoPago
{
    void Procesar(decimal monto);
    string ObtenerDescripcion();
}

public class PagoEfectivo : IMetodoPago
{
    public void Procesar(decimal monto)
        => Console.WriteLine($"Efectivo: ${monto}");
    public string ObtenerDescripcion() => "Efectivo";
}

public class PagoTarjeta : IMetodoPago
{
    private string _numero;
    public PagoTarjeta(string numero) 
        => _numero = numero;
    public void Procesar(decimal monto)
        => Console.WriteLine($"Tarjeta: ${monto}");
    public string ObtenerDescripcion() => "Tarjeta";
}

// El cliente no sabe ni le importa qué tipo es
public class Caja
{
    public void Cobrar(IMetodoPago metodo, decimal m)
        => metodo.Procesar(m); // ← polimorfismo
}
Polymorphism GRASP + Open/Closed SOLID = agregar un nuevo tipo de pago solo requiere crear una nueva clase. Cero modificaciones al código existente.
GRASP · 07 + 08 + 09

Pure Fabrication, Indirection & Protected Variations

Los tres patrones avanzados de GRASP

07 Pure Fabrication
Clase inventada (no del dominio) para lograr bajo acoplamiento y alta cohesión.
Ejemplo: EmailService, LoggerService, RepositorioPedidoSQL. No representan conceptos del mundo real, pero resuelven necesidades técnicas de forma limpia.
// Pure Fabrication:
// EmailService no existe en el dominio
// del negocio, pero es necesaria
public class EmailService
{
    public void Notificar(string to, 
                          string msg) { }
}
08 Indirection
Introducir un intermediario entre dos componentes para desacoplarlos.
Ejemplo: En lugar de que el Controller llame directamente a la DB, introduce un Repositorio como intermediario.
// Indirection: Repositorio como
// intermediario entre Service y DB
public class PedidoController
{
    // No habla con la DB directo
    private readonly IPedidoRepo _repo;
    public PedidoController(IPedidoRepo r)
        => _repo = r;
}
09 Protected Variations
Envolvé los puntos de variación en una interfaz estable para proteger al resto del sistema.
Ejemplo: Si la forma de calcular impuestos varía (IVA, IRPF, exento), encapsulala detrás de ICalculadorImpuesto.
// Protected Variations
public interface ICalculadorImpuesto
{
    decimal Calcular(decimal base);
}
// Las variaciones están protegidas
// detrás de la interfaz

Los 5 Principios SOLID

Robert C. Martin (Uncle Bob) — cinco principios para código orientado a objetos de calidad

S
Single Responsibility
Una razón para cambiar
O
Open/Closed
Abierto a extensión, cerrado a modificación
L
Liskov Substitution
Subtipos reemplazables
I
Interface Segregation
Interfaces específicas
D
Dependency Inversion
Depender de abstracciones
¿Cuándo aplicarlos?
No son dogma — son guías. En proyectos pequeños, SOLID puede ser overkill. En sistemas que van a crecer y cambiar, son esenciales para mantener el control.
SOLID y GRASP juntos
GRASP te dice dónde poner cada responsabilidad. SOLID te dice cómo estructurarla. Se complementan perfectamente.

S
SOLID · S

Single Responsibility Principle

"Una clase debe tener una, y solo una, razón para cambiar"

VIOLA SRP — múltiples responsabilidades
// ❌ Esta clase tiene 3 razones para cambiar:
// 1. Si cambia el formato del reporte
// 2. Si cambia la DB
// 3. Si cambia el formato del email
public class Empleado
{
    public string Nombre { get; set; }
    public decimal Salario { get; set; }

    // Razón 1: lógica de negocio
    public decimal CalcularSueldoNeto() { ... }

    // Razón 2: persistencia
    public void GuardarEnDB() { ... }

    // Razón 3: presentación/comunicación
    public void EnviarReportePorEmail() { ... }
}
CUMPLE SRP
// ✅ Cada clase tiene UNA responsabilidad

// Solo datos y lógica de dominio
public class Empleado
{
    public string Nombre { get; set; }
    public decimal Salario { get; set; }
    public decimal CalcularSueldoNeto() { ... }
}

// Solo persistencia
public class EmpleadoRepositorio
{
    public void Guardar(Empleado emp) { ... }
    public Empleado? ObtenerPorId(int id) { ... }
}

// Solo comunicación
public class EmpleadoNotificador
{
    private readonly IEmailService _email;
    public void NotificarSueldo(Empleado emp)
        => _email.Enviar(emp.Email, "...");
}
Truco: Si necesitás "y" para describir lo que hace la clase (persiste Y notifica Y calcula), viola SRP.
O
SOLID · O

Open/Closed Principle

"Abierto para extensión, cerrado para modificación"

VIOLA OCP — switch que crece
// ❌ Cada nuevo tipo de descuento
// obliga a modificar esta clase
public class CalculadorDescuento
{
    public decimal Calcular(
        string tipo, decimal precio)
    {
        switch (tipo)
        {
            case "navidad": return precio * 0.8m;
            case "verano":  return precio * 0.9m;
            // ← agregar aquí cada vez ← MAL
            default: return precio;
        }
    }
}
CUMPLE OCP — extensión por herencia/interfaz
// ✅ Interfaz cerrada — no se modifica
public interface IDescuento
{
    decimal Aplicar(decimal precio);
}

// Extensiones abiertas — nuevas clases
public class DescuentoNavidad : IDescuento
{
    public decimal Aplicar(decimal p) => p * 0.80m;
}

public class DescuentoVerano : IDescuento
{
    public decimal Aplicar(decimal p) => p * 0.90m;
}

// ✅ Para agregar "BlackFriday":
// Solo crear nueva clase — cero modificaciones
public class DescuentoBlackFriday : IDescuento
{
    public decimal Aplicar(decimal p) => p * 0.50m;
}

// Uso — no cambia nunca
public class Carrito
{
    public decimal CalcularTotal(
        decimal precio, IDescuento descuento)
        => descuento.Aplicar(precio);
}
OCP se logra con los mismos mecanismos que Polymorphism GRASP: interfaces o clases abstractas. Son la misma solución vista desde ángulos distintos.
L
SOLID · L

Liskov Substitution Principle

"Si S hereda de T, podés usar S donde se esperaba T, sin sorpresas"

La prueba de Liskov: si reemplazás el padre por el hijo, ¿el programa sigue funcionando igual?
VIOLA LSP — el hijo rompe el contrato
public class Rectangulo
{
    public virtual double Ancho  { get; set; }
    public virtual double Alto   { get; set; }
    public double Area() => Ancho * Alto;
}

// ❌ Cuadrado viola el contrato del Rectángulo
public class Cuadrado : Rectangulo
{
    public override double Ancho
    {
        set { base.Ancho = value; base.Alto = value; }
    }
}

// Este código falla si le pasás un Cuadrado:
void TestArea(Rectangulo r)
{
    r.Ancho = 4; r.Alto = 5;
    // Esperamos 20, pero Cuadrado da 25 ← bug
    Assert.Equal(20, r.Area());
}
CUMPLE LSP — jerarquía correcta
// ✅ Forma base sin relación problemática
public abstract class Forma
{
    public abstract double Area();
}

public class Rectangulo : Forma
{
    public double Ancho { get; set; }
    public double Alto  { get; set; }
    public override double Area() => Ancho * Alto;
}

public class Cuadrado : Forma
{
    public double Lado { get; set; }
    public override double Area() => Lado * Lado;
}

// Ahora SÍ podés usar cualquier Forma
void ImprimirArea(Forma f)
{
    Console.WriteLine($"Área: {f.Area()}");
    // ✅ funciona con Rectangulo y Cuadrado
}
LSP no dice "nunca heredes". Dice que la herencia debe representar una relación ES-UN real y no romper el comportamiento esperado.

SOLID · I

Interface Segregation Principle

"Los clientes no deberían verse forzados a implementar interfaces que no usan"

VIOLA ISP — interfaz gorda
// ❌ Interfaz que hace todo
public interface IImpresora
{
    void Imprimir(Documento doc);
    void Escanear(Documento doc);
    void Faxear(Documento doc);
    void EnviarEmail(Documento doc);
}

// ❌ Una impresora básica implementa
// cosas que no puede hacer
public class ImpresoraBasica : IImpresora
{
    public void Imprimir(Documento doc) { ... }
    public void Escanear(Documento doc)
        => throw new NotImplementedException();
    public void Faxear(Documento doc)
        => throw new NotImplementedException();
    // ← código muerto / trampa
}
CUMPLE ISP — interfaces específicas
// ✅ Interfaces pequeñas y específicas
public interface IImprimible
{
    void Imprimir(Documento doc);
}

public interface IEscaneable
{
    void Escanear(Documento doc);
}

public interface IFaxeable
{
    void Faxear(Documento doc);
}

// Impresora básica: solo lo que puede hacer
public class ImpresoraBasica : IImprimible
{
    public void Imprimir(Documento doc) { ... }
}

// Multifunción: implementa lo que corresponde
public class ImpresoraMultifuncion
    : IImprimible, IEscaneable, IFaxeable
{
    public void Imprimir(Documento doc) { ... }
    public void Escanear(Documento doc) { ... }
    public void Faxear(Documento doc)   { ... }
}
Si tu interfaz tiene métodos que algunos implementadores dejan vacíos o con NotImplementedException, es señal de que viola ISP.
D
SOLID · D

Dependency Inversion Principle

"Depende de abstracciones, no de concreciones"

Dos reglas:
1. Los módulos de alto nivel no deben depender de módulos de bajo nivel. Ambos deben depender de abstracciones.
2. Las abstracciones no deben depender de detalles. Los detalles deben depender de abstracciones.
VIOLA DIP — depende del concreto
// ❌ Alto nivel depende de bajo nivel
public class PedidoServicio
{
    // ← acoplado a la implementación SQL
    private readonly SqlPedidoRepositorio _repo
        = new SqlPedidoRepositorio();

    public Pedido ObtenerPedido(int id)
        => _repo.ObtenerPorId(id);

    // Si cambiamos a MongoDB... hay que
    // modificar PedidoServicio. ← MAL
}
CUMPLE DIP — inversión de dependencias
// ✅ Abstracción en el medio
public interface IPedidoRepositorio
{
    Pedido? ObtenerPorId(int id);
    void Guardar(Pedido pedido);
}

// Alto nivel depende de la abstracción
public class PedidoServicio
{
    private readonly IPedidoRepositorio _repo;

    // La dependencia se INYECTA
    public PedidoServicio(IPedidoRepositorio repo)
        => _repo = repo;

    public Pedido ObtenerPedido(int id)
        => _repo.ObtenerPorId(id) 
           ?? throw new KeyNotFoundException();
}

// Bajo nivel implementa la abstracción
public class SqlPedidoRepositorio 
    : IPedidoRepositorio { ... }

public class MongoDBPedidoRepositorio 
    : IPedidoRepositorio { ... }

// En Program.cs (o DI container):
// services.AddScoped<IPedidoRepositorio,
//                     SqlPedidoRepositorio>();
Relaciones

Cómo se conectan

GRASP y SOLID no son rivales — se refuerzan mutuamente

GRASP → SOLID
HC
High Cohesion SRP
Una clase con alta cohesión hace una sola cosa → tiene una sola razón para cambiar (SRP).
Po
Polymorphism GRASP OCP
Reemplazar condicionales por polimorfismo hace el código abierto a extensión sin modificación.
PV
Protected Variations OCP
Encapsular lo que varía detrás de interfaces es exactamente el mecanismo del OCP.
In
Indirection DIP
Introducir un intermediario (interfaz) entre capas es la base del Dependency Inversion.
LC
Low Coupling DIP + ISP
Depender de interfaces pequeñas (ISP) reduce el acoplamiento y permite la inversión de dependencias.

Sistema de Notificaciones

Aplicá GRASP + SOLID en un ejercicio real — identificá qué principio viola cada fragmento

Este sistema tiene múltiples problemas de diseño. Identificalos y refactorizalo:

NotificadorMal.cs — lleno de problemas 
// ⚠ Este código viola varios principios.
// Identificá cuáles antes de ver la solución.

public class Notificador
{
    // Crea dependencias directamente
    private SmtpClient _smtp = new SmtpClient();
    private TwilioClient _twilio = new TwilioClient();

    public void Notificar(
        string tipo, string usuario, string mensaje)
    {
        // Condicional por tipo de notificación
        if (tipo == "email")
        {
            _smtp.Send(usuario, mensaje);
            Console.WriteLine("Enviado por email");
            // También persiste en la misma clase
            File.AppendAllText("log.txt", mensaje);
        }
        else if (tipo == "sms")
        {
            _twilio.Send(usuario, mensaje);
            File.AppendAllText("log.txt", mensaje);
        }
        // ¿y si agregamos WhatsApp, Push, Slack?
    }
}
PROBLEMAS IDENTIFICADOS
S
Viola SRP
La clase notifica, hace logging y configura transporte — tres responsabilidades.
O
Viola OCP
Agregar WhatsApp requiere modificar el if/else existente.
D
Viola DIP
new SmtpClient() y new TwilioClient() directamente — depende de concreciones.
Po
Viola Polymorphism GRASP
if/else por tipo en lugar de polimorfismo.
