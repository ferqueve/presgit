Introducción
Dos tipos de relaciones entre clases
«es un» (is-a)

Clasificación taxonómica. Una clase es un tipo de otra clase más abstracta.

  • Un puma es un mamífero
  • Un mamífero es un animal
  • Un CD es un Disco
  • Un DVD es un Disco
→ Herencia / Generalización
«es parte de» (has-a)

Relación todo-parte. Un objeto está compuesto por otros objetos.

  • Un auto tiene un motor
  • Una discoteca tiene discos
  • Un edificio tiene departamentos
  • Una orden tiene ítems
→ Agregación / Composición

Ambas relaciones crean una ilusión de simplicidad: trabajamos con el todo sin preocuparnos por los detalles de las partes o de la jerarquía.

Herencia
Generalización y Especialización
Definición

La generalización es una relación taxonómica entre una clase base más general o abstracta y otra clase sucesora más específica o concreta. La clase base es completamente consistente con la clase sucesora.

Clase Base / Superclase

Clase más abstracta y general. También llamada: clase ancestral, superclase, clase madre.

Clase Sucesora / Subclase

Clase más concreta y específica. También llamada: clase derivada, subclase, clase hija.

JERARQUÍA DE EJEMPLO
Herencia y Reutilización

La herencia permite crear nuevas clases a partir de clases existentes. Una subclase hereda atributos y métodos de su superclase y habitualmente agrega nuevos. Importante: no usar la herencia solo por reutilización; las clases también deben estar en relación taxonómica real.

Herencia en C#
La clase Disk

Tengo una discoteca con CD de música y DVD de películas. Ambos son discos: tienen nombre, género y año. Extraemos esas características comunes a una clase base.

Disk.cs public class Disk { protected Disk(string name) { this.Name = name; } public string Name { get; set; } public string Genre { get; set; } public int Year { get; set; } }
CD.cs public class CD : Disk // CD "extiende" Disk { private List<string> songs = new(); public CD(string name, string[] songs) : base(name) // llama ctor de Disk { this.songs.AddRange(songs); } public string Player { get; set; } public IEnumerable<string> Songs => songs; }
Sintaxis C#
  • CD : Disk — CD hereda de Disk
  • : base(name) — llama al constructor de la superclase
  • CD hereda Name, Genre, Year sin redeclararlos
  • CD agrega Player y Songs
DVD.cs public class DVD : Disk { public DVD(string name) : base(name) { } public string Director { get; set; } public List<string> Cast = new(); public bool IsActorInCast(string a) => Cast.Contains(a); }

Herencia múltiple — Cuando una clase sucesora puede tener más de una clase base. C# no soporta herencia múltiple de clases, pero sí de interfaces.

Herencia avanzada
Clases y métodos abstractos

Queremos que todo Disk pueda decir si es favorito, pero el criterio es diferente para CD (por intérprete) y DVD (por género, director y actores). Usamos métodos abstractos para obligar a las subclases a implementar el método.

Disk.cs — abstracta // No se puede instanciar Disk directamente public abstract class Disk { protected Disk(string name) { this.Name = name; } public string Name { get; set; } public string Genre { get; set; } public int Year { get; set; } // Sin implementación — las subclases DEBEN // proveer una implementación con override public abstract bool IsFavorite(); }
Reglas clave
  • Una clase con al menos un método abstracto es clase abstracta
  • No se pueden crear instancias de clases abstractas
  • Las subclases deben implementar todos los métodos abstractos (o también ser abstractas)
  • En C#: abstract para declarar, override para implementar
Métodos virtuales

Un método virtual tiene implementación pero puede ser sobreescrito. Un método abstract no tiene implementación y debe ser sobreescrito.

CD.cs — override public class CD : Disk { private List<string> favPlayers = new() { "Carlos Santana", "Pink Floyd" }; public override bool IsFavorite() => favPlayers.Contains(this.Player); }
DVD.cs — override public class DVD : Disk { public string FavActor = "John Travolta"; public string FavDir = "Tarantino"; public string FavGenre = "Acción"; public override bool IsFavorite() => (IsActorInCast(FavActor) || Director.Equals(FavDir)) && Genre.Equals(FavGenre); }
Agregación
Relación todo-parte
Definición

La agregación es una relación todo-parte entre una clase compuesta y una clase componente. Las instancias del componente suelen existir independientemente de la existencia del compuesto.

DiskLibrary contiene Disks

Una DiskLibrary puede contener 0 o muchos Disk. Un Disk puede estar en 0 o 1 DiskLibrary.

Diagrama UML — Agregación

Diamante vacío = Agregación

DiskLibrary.cs public class DiskLibrary { private IList<Disk> discs = new List<Disk>(); public void AddDisc(Disk disc) { discs.Add(disc); } public IList<Disk> GetDiscsByGenre(string genre) => discs.Where(d => d.Genre.Equals(genre, StringComparison.OrdinalIgnoreCase)).ToList(); public IList<Disk> GetFavorites() // ← usa polimorfismo => discs.Where(d => d.IsFavorite()).ToList(); }

Noten que GetFavorites() usa polimorfismo: llama d.IsFavorite() sin saber si d es un CD o un DVD. El encadenamiento dinámico resuelve qué método ejecutar en tiempo de ejecución.

Polimorfismo
Encadenamiento dinámico vs estático
Encadenamiento Dinámico (Tardío)

El método a ejecutar se determina en tiempo de ejecución buscando en la clase del objeto receptor y en sus ancestras.

  • Necesario para el polimorfismo
  • Métodos virtual y abstract en C#
  • Genera código que busca el método en runtime
  • Ligero overhead de ejecución
Encadenamiento Estático (Temprano)

El método a ejecutar se determina en tiempo de compilación según el tipo declarado de la variable.

  • Default en C# para métodos no virtuales
  • Mayor rendimiento
  • Certeza de qué código se ejecuta
  • No permite polimorfismo real
polimorfismo en acción Disk d1 = new CD("The Wall", new[] { "Another Brick" }) { Player = "Pink Floyd" }; Disk d2 = new DVD("Pulp Fiction") { Director = "Tarantino", Genre = "Acción" }; // ¿Qué IsFavorite() se llama? Depende del TIPO REAL del objeto (runtime) Console.WriteLine(d1.IsFavorite()); // Ejecuta CD.IsFavorite() → true Console.WriteLine(d2.IsFavorite()); // Ejecuta DVD.IsFavorite() → true

Downcast / Typecast — Evitarlo cuando sea posible:

// ⚠️ Evitar: preguntar la clase rompe el polimorfismo if (disc is CD) { ((CD)disc).Player ... } // ✅ Mejor: aprovechar el polimorfismo con métodos virtuales disc.IsFavorite(); // sin preguntar la clase
Diseño OO
Composición vs Herencia

La composición y delegación es una alternativa a la herencia para la reutilización. En el contexto de la reutilización, toda implementación que use herencia se puede cambiar por una equivalente que use composición.

Aspecto Composición Herencia
VisibilidadCaja negra — no requiere conocer la implementación internaCaja blanca — requiere conocer los detalles internos
MomentoDinámica — puede cambiar en tiempo de ejecuciónEstática — se define en tiempo de compilación
ImplementaciónPor código, requiere código adicional del programadorDeclarativa, una palabra clave basta
Cantidad de códigoMás código (mayor posibilidad de errores)Menos código
ReutilizaciónSelectiva — elige qué reutilizarTodo o nada — hereda todo
ClasesPuede reutilizar 1 o más simultáneamenteSimple: 1; Múltiple: 2+ (poco soportada)
TiposSin relación impuesta entre los tiposImpone relación supertipo/subtipo
composición + delegación // En vez de: class CD : Disk { ... } // Composición: CD "tiene" un Disk public class CD { private readonly Disk _disk; // ← referencia al componente public CD(Disk disk) { _disk = disk; } // Delegación: CD delega en _disk public string Name => _disk.Name; public string Genre => _disk.Genre; // Comportamiento propio de CD public string Player { get; set; } }
