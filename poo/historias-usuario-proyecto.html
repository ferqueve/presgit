Las Historias de Usuario del proyecto

16 historias de usuario que definen qué debe hacer el sistema de recomendaciones para un chatbot.

Hoy vamos a recorrerlas juntos: qué significan, qué pide cada una y cómo se relacionan entre sí.

¿Qué necesita el usuario?

Antes de escribir código, hay que entender para quién y para qué.

Conceptos

¿Qué es una Historia de Usuario?

Formato estándar

Como [tipo de usuario]
quiero [hacer algo]
para [obtener un beneficio].

Ejemplo concreto

Como usuario
quiero registrarme
para poder utilizar el sistema.

¿Para qué sirven?

Describen funcionalidad desde el punto de vista del usuario.
No dicen cómo implementar, sino qué se necesita.
Cada historia tiene criterios de aceptación: condiciones que deben cumplirse para considerarla terminada.
Son la base para los casos de prueba.
En la segunda entrega, cada historia deberá estar implementada y validada con casos de prueba usando la fachada.

HU1

Registrarme en el sistema

Como usuario quiero registrarme para poder utilizar el sistema.

Criterios de aceptación
Ingresando mis datos se crea una cuenta válida.
El sistema valida que el usuario no exista previamente.
El usuario queda habilitado para usar el sistema.
¿Qué implica en el código?

El sistema necesita una lista (o repositorio) de usuarios y un método para agregar uno nuevo.

// En la fachada o repositorio: bool RegistrarUsuario(string nombre, string email); // Antes de registrar, verificar: bool yaExiste = BuscarPorEmail(email) != null; if (yaExiste) return false;
No se puede registrar dos veces el mismo usuario. Definir por cuál campo se identifica (email, nombre, etc.).
HU2 · HU10

Preferencias e Interacciones

Dos formas de enriquecer el perfil de un usuario.

HU2
Definir preferencias

El usuario selecciona atributos que le interesan: género musical, categoría, temática, etc.

Puede elegir uno o más atributos.
Las preferencias quedan almacenadas.
Las recomendaciones las tienen en cuenta.
// Ejemplo: usuario.Preferencias = new List<string> { "rock", "pop" };
HU10
Registrar interacciones

Cada vez que el usuario consume un ítem (reproduce, visualiza, compra), eso se guarda en su historial.

Cada consumo genera una interacción.
Las interacciones se almacenan en el historial.
El sistema usa ese historial para recomendar.
// Ejemplo: usuario.HistorialIds.Add(cancionId);
Diferencia clave: las preferencias las declara el usuario; las interacciones se generan automáticamente con el uso.
HU3 · HU9

Ver y ordenar recomendaciones

La lista final que el usuario recibe siempre debe estar ordenada.

HU3
Ver recomendaciones

El sistema presenta una lista de ítems sugeridos con información básica de cada uno.

Muestra una lista de ítems recomendados.
Tiene al menos un criterio de orden.
Cada ítem muestra su nombre y atributos relevantes.
HU9
Ordenar por criterio

Las recomendaciones no se muestran en cualquier orden: el sistema define cómo rankearlas.

Existe un criterio de orden definido (score, popularidad…).
Los resultados se presentan ordenados.
Cambiar el criterio modifica el orden.
Posibles criterios de ordenamiento
Score de coincidencia Popularidad (más consumido) Más reciente Orden alfabético
HU4 · HU5

Estrategias de recomendación (1/3)

Dos de las formas en que el sistema decide qué recomendar.

HU4
Por historial

El sistema analiza lo que el usuario consumió y sugiere contenido parecido a eso.

Usa las interacciones previas del usuario.
Genera sugerencias distintas a lo ya consumido.
Cambiar el historial afecta las recomendaciones.
Ej.: si escuché muchas canciones de rock, el sistema infiere que me gusta el rock.
HU5
Por preferencias declaradas

El sistema compara los atributos de los ítems contra lo que el usuario dijo que le gusta.

Usa los atributos definidos en preferencias.
Las recomendaciones coinciden con al menos una preferencia.
Cambiar preferencias modifica los resultados.
Es la estrategia más sencilla de implementar: comparar listas de strings.
HU6 · HU7

Estrategias de recomendación (2/3)

Para usuarios con historial vs. usuarios nuevos.

HU6
Usuarios similares

Filtrado colaborativo: "si a personas parecidas a mí les gustó X, probablemente a mí también."

Identifica usuarios con comportamiento similar.
Recomienda ítems consumidos por esos usuarios.
Excluye lo que el usuario ya consumió.
¿Cómo definir "similar"? Por ejemplo: usuarios que compartieron al menos N ítems en su historial.
HU7
Popularidad / Tendencias

Para un usuario nuevo que no tiene historial ni preferencias, el sistema no puede "fallar": usa lo más popular.

Si el usuario no tiene historial, usa una estrategia alternativa.
Muestra ítems populares o recientes.
El sistema no falla ante ausencia de datos.
Este caso se llama cold start. Hay que contemplarlo siempre.
HU15

Estrategias de recomendación (3/3)

Contenido relacionado a algo que ya consumí.

HU15
Recomendaciones relacionadas

"Escuché esta canción y quiero más canciones parecidas a esta."

El sistema identifica los atributos del ítem base.
Recomienda ítems con atributos similares.
El resultado cambia según el ítem seleccionado.
Resumen de las 5 estrategias
Por preferencias (HU5) Atributos vs. gustos declarados.
Por historial (HU4) Basada en lo ya consumido.
Colaborativa (HU6) Usuarios similares.
Popularidad (HU7) Para cold start.
Relacionadas (HU15) Parecidas a un ítem puntual.
El motor debe poder cambiar de estrategia sin modificar su estructura interna. ¡Piensen en el patrón Strategy!
HU8 · HU13

Filtros sobre las recomendaciones

No todo lo que genera el motor llega al usuario: hay filtros que depuran la lista.

HU8
Filtrar por atributo

El usuario puede decir "no me muestres contenido de X género" o "excluí este tipo de ítem".

El usuario define filtros por atributo.
Los ítems que cumplen el filtro son excluidos.
El sistema aplica los filtros antes de mostrar.
// Ejemplo: excluir género "reggaeton" if (item.Atributos.Contains("reggaeton")) continue;
HU13
No repetir consumidos

Nunca se debe recomendar algo que el usuario ya consumió. Este filtro es automático y siempre activo.

Excluye ítems del historial del usuario.
No aparecen duplicados en recomendaciones.
El filtro se aplica automáticamente.
// Ya implementado en el motor base: if (usuario.HistorialIds.Contains(item.Id)) continue;
Los filtros se aplican después de generar las recomendaciones y antes de ordenarlas. El orden lógico es: Generar → Filtrar → Ordenar → Mostrar.
HU11 · HU12

Valoraciones e historial visible

El usuario tiene control sobre lo que le gustó y puede ver lo que ya consumió.

HU11
Me gusta / No me gusta

El usuario puede indicar su valoración de un ítem. Esto es distinto a simplemente haberlo consumido.

Puede marcar un ítem como "me gusta" o "no me gusta".
Esta información afecta futuras recomendaciones.
El sistema diferencia consumo de valoración.
Una Interaccion podría tener un campo tipo con valores como: REPRODUCCION, LIKE, DISLIKE.
HU12
Ver mi historial

El usuario puede consultar en cualquier momento los ítems que ya consumió.

El usuario puede consultar su historial.
El historial muestra los ítems consumidos.
Se actualiza automáticamente con cada consumo.
// Obtener historial del usuario: List<IRecomendable> ObtenerHistorial(Usuario u, List<IRecomendable> catalogo);
HU14 · HU16

Guardados y administración

Funciones adicionales para el usuario y para el administrador del sistema.

HU14
Guardar para después

El usuario puede marcar ítems para verlos más tarde, sin consumirlos todavía.

El usuario puede guardar ítems.
Existe una lista de "guardados".
Los ítems guardados pueden recuperarse.
Guardar un ítem no equivale a consumirlo: no debe afectar el historial ni las recomendaciones de la misma forma que una reproducción.
HU16
Admin: eliminar ítems

Un administrador puede dar de baja ítems del catálogo sin romper el sistema.

Se pueden eliminar ítems existentes.
Los ítems eliminados no aparecen en recomendaciones.
No se rompe el sistema si el ítem fue consumido antes.
Si un ítem eliminado está en el historial de alguien, el sistema debe manejarlo sin errores (ej: ignorarlo o marcarlo como no disponible).
Requisito clave de diseño

El motor debe estar desacoplado del dominio

Este es el requisito de diseño más importante del proyecto: el recomendador no puede depender de clases concretas.

¿Qué significa?

No debe depender de Cancion, Pelicula o Producto.
Debe trabajar con una abstracción: una interfaz como IRecomendable.
El mismo motor puede usarse para cualquier dominio sin modificarlo.
Agregar un nuevo tipo de ítem no requiere tocar el motor.
Acoplado (mal diseño)
public class Recomendador { // Solo sirve para canciones ❌ public List<Cancion> Recomendar( List<Cancion> canciones) { } }
Desacoplado (buen diseño)
public class Recomendador { // Sirve para cualquier dominio ✅ public List<IRecomendable> Recomendar( List<IRecomendable> items) { } }
Principio de diseño detrás: Dependency Inversion — depender de abstracciones, no de implementaciones concretas.
