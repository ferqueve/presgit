SOLID · Principio 4

Interface
Segregation
Principle

Diseño orientado a objetos robusto y desacoplado: que cada cliente dependa solo de lo que realmente usa.

C# / .NET Robert C. Martin FIT – UCU

ISP forma parte del acrónimo SOLID, cinco principios de diseño OO para hacer el software más comprensible, flexible y mantenible.

Motivación

¿Por qué surge ISP?

A veces una clase expone más responsabilidades de las que sus colaboradores necesitan. Esto genera una dependencia "todo o nada" problemática.

El problema
  • !
    Una clase tiene más operaciones de las que un cliente usa.
  • !
    El cliente depende del tipo completo, aunque solo use 1 método.
  • !
    Un cambio en algo que no le incumbe lo puede afectar.
La solución
  • 1
    Identificar grupos de operaciones que distintos clientes usan.
  • 2
    Separar esas operaciones en interfaces específicas.
  • 3
    Cada cliente depende solo del tipo que necesita.
Recuerda: una clase tiene múltiples tipos → su propio tipo + ancestros + interfaces que implementa.
Definición

El Enunciado

Los clientes no deben ser forzados a depender de tipos que no usan.

— Robert C. Martin
Cliente

Objeto que pide colaboración y usa responsabilidades de otra clase.

Tipo

En C#: la clase propia + sus ancestros + las interfaces que implementa.

Segregar

Separar interfaces grandes en partes cohesivas y enfocadas.

Nota terminológica: Martin habla de "interfaces" pero en C# esa palabra tiene un significado específico de construcción del lenguaje. Por eso usamos el término más general tipo.

Diagrama

El problema en un diagrama

Las clases A y B dependen de Interfaz C, que define O1 y O2. A solo usa O1 y B solo O2. Un cambio en O1 afecta también a B.

Aunque B no usa O1, sí depende del tipo que la declara. Si O1 cambia, B podría recompilarse o verse afectada innecesariamente.

Solución

La solución ISP

Separamos las operaciones O1 y O2 en dos interfaces distintas. Ahora cada cliente depende solo del tipo que usa.

Un cambio en O1 (Interfaz C1) solo afecta a Clase A. Clase B queda completamente aislada.
Ejemplo

Caso de estudio: Punto de Venta

La clase Sale representa un ticket de venta. Cuando la venta se cierra, no se pueden agregar ni quitar ítems (IsClosed).

Sale.cs
public class Sale
{
    public bool IsClosed { get; private set; }

    public SalesLineItem AddLineItem(double quantity, ProductSpecification product)
    {
        if (this.IsClosed)
            throw new InvalidOperationException("La venta está cerrada.");

        SalesLineItem item = new SalesLineItem(quantity, product);
        this.lineItems.Add(item);
        return item;
    }

    public void RemoveLineItem(SalesLineItem item)
    {
        if (this.IsClosed)
            throw new InvalidOperationException("La venta está cerrada.");
        this.lineItems.Remove(item);
    }

    public void Close()
    {
        if (this.IsClosed) throw new Exception("La venta ya está cerrada.");
        this.IsClosed = true;
    }

    public void Open()
    {
        if (!this.IsClosed) throw new Exception("La venta no está cerrada.");
        this.IsClosed = false;
    }
}

El siguiente desafío: cerrar la venta automáticamente si queda abierta más tiempo del necesario → necesitamos un temporizador.

Mecanismo

CountdownTimer e ITimerClient

Para el cierre automático usamos CountdownTimer: un objeto registrado como ITimerClient recibe un aviso TimeOut() cuando el tiempo expira.

ITimerClient.cs
public interface ITimerClient
{
    void TimeOut();
}
CountdownTimer.cs
public class CountdownTimer
{
    private ITimerClient client;
    private Timer timer;

    public void Register(int timeOut, ITimerClient client)
    {
        this.client = client;
        this.timer = new Timer(
            this.OnTimedEvent, null,
            timeOut, Timeout.Infinite);
    }

    private void OnTimedEvent(object state)
    {
        this.client.TimeOut();
        this.timer.Change(
            Timeout.Infinite, Timeout.Infinite);
    }
}
Para que Sale reciba el aviso, debe implementar ITimerClient… pero esto trae un problema de ISP.
Violación

Sale implementa ITimerClient → problema

Al hacer que Sale implemente ITimerClient, todas las clases que usan Sale (como Program) pasan a depender indirectamente de ITimerClient.

Consecuencia: Si cambia la firma de TimeOut() en ITimerClient (ej.: agregar timeOutId), clases como Program o ConsolePrinter quedan afectadas sin razón.

Evolución

El cambio que evidencia el problema

Se agrega un timeOutId para evitar cierres múltiples accidentales. Esto modifica la firma de TimeOut() → todas las implementaciones deben cambiar.

ITimerClient.cs – modificado
// ANTES:
public interface ITimerClient
{
    void TimeOut();           // ← versión original
}

// DESPUÉS (con identificador único):
public interface ITimerClient
{
    void TimeOut(object timeOutId);   // ← Sale TAMBIÉN debe cambiar
}

// CountdownTimer también cambia:
public void Register(object timeOutId, int timeOut, ITimerClient client)
{
    this.client = client;
    // timeOutId se pasa al Timer y se reenvía en TimeOut()
    this.timer = new Timer(this.OnTimedEvent, timeOutId, timeOut, Timeout.Infinite);
}
Impacto negativo

Clases que ni usan el timer (Program, ConsolePrinter) se ven obligadas a recompilarse por culpa de la dependencia en ITimerClient a través de Sale.

Lo que necesitamos

Eliminar la implementación directa de ITimerClient en Sale. Patrón a usar: Adapter.

Solución

Patrón Adapter al rescate

Creamos TimerAdapter: una clase que implementa ITimerClient y delega a Sale. Sale ya no implementa ITimerClient directamente.

El ◆ diamante indica composición: Sale contiene una instancia de TimerAdapter. El adaptador es una clase anidada privada → invisible desde fuera.
Implementación

Código del TimerAdapter

TimerAdapter se declara como clase anidada privada dentro de Sale, lo que la hace invisible desde el exterior.

Sale.cs – versión final con Adapter
public class Sale  // Ya NO implementa ITimerClient
{
    private CountdownTimer timer = new CountdownTimer();
    private TimerAdapter timerClient;

    public Sale()
    {
        this.StartCountdown();
    }

    public void Open()
    {
        if (!this.IsClosed) throw new Exception("La venta no está cerrada.");
        this.IsClosed = false;
        this.StartCountdown();   // reinicia la cuenta regresiva
    }

    private void StartCountdown()
    {
        this.timerClient = new TimerAdapter(this);
        this.timer.Register(1000, this.timerClient);
    }

    // ── Clase anidada privada ──────────────────────────────────────
    private class TimerAdapter : ITimerClient
    {
        private Sale sale;

        public TimerAdapter(Sale sale)
        {
            this.sale = sale;
        }

        public void TimeOut()
        {
            this.sale.Close();   // delega el cierre a Sale
        }
    }
}
Resultado

Ventajas del diseño final

Sale limpia

Sale no implementa ITimerClient. Sus clientes (Program, ConsolePrinter) no tienen dependencia en ese tipo. Cambios en ITimerClient no los afectan.

Encapsulamiento total

TimerAdapter es una clase anidada privada. Solo existe dentro de Sale. Nadie más conoce su existencia.

Evolución independiente

Si ITimerClient cambia nuevamente, solo se modifica TimerAdapter. El tipo público de Sale no cambia.

ISP + Adapter

ISP identifica el problema; el patrón Adapter provee la solución estructural. Ambos se complementan perfectamente.

ISP promueve interfaces pequeñas, cohesivas y orientadas al cliente.
