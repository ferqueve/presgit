Definición

¿Qué es una excepción?

Las aplicaciones necesitan controlar errores en tiempo de ejecución de forma coherente. .NET proporciona un modelo uniforme: las operaciones indican errores produciendo excepciones.

Definición formal
Una excepción es cualquier condición de error o comportamiento inesperado que encuentra un programa en ejecución.

En .NET, una excepción es un objeto que hereda de System.Exception. Se lanza desde la sección de código donde ocurrió el problema y se pasa hacia arriba en la pila hasta que la aplicación la controla o el programa finaliza.

Fuentes de excepciones
  • 1
    Error en el código propio.
  • 2
    Error en código de librería compartida.
  • 3
    Recursos del S.O. no disponibles.
  • 4
    Condiciones inesperadas del runtime.
Algunas excepciones son recuperables (aplicación puede continuar).
Las excepciones de runtime generalmente no son recuperables.
Flujo

Cómo viaja una excepción

Una excepción lanzada sube por la pila de llamadas buscando un manejador. Si no lo encuentra, el programa termina.

Si ningún frame de la pila tiene un catch adecuado, el runtime finaliza la aplicación.
Comparación

Excepciones vs. códigos de retorno

El modelo tradicional usaba valores de retorno o variables globales para indicar errores. Las excepciones son superiores en varios aspectos.

Códigos de retorno Excepciones .NET
Los errores pueden pasar desapercibidos: nadie fuerza a verificar el código de retorno. Si no se captura la excepción, el runtime termina la app. El error no se ignora.
Valores inválidos continúan propagándose si no se verifica el código de retorno. Los valores inválidos no se propagan: la excepción interrumpe el flujo inmediatamente.
El manejo de errores mezcla lógica de negocio con lógica de error. El código de manejo de errores está separado en bloques catch.
Cada lenguaje o librería usa convenciones distintas para los códigos. El modelo funciona igual en todos los lenguajes .NET (C#, VB, F#…).
No cruzan límites de proceso o máquina automáticamente. Las excepciones pueden iniciarse en varios procesos e incluso entre máquinas.
El código de control de excepciones puede agregarse a una aplicación existente para aumentar su confiabilidad sin cambiar la lógica de negocio.
Jerarquía

Jerarquía de System.Exception

Todas las excepciones en .NET heredan de System.Exception. Conocer la jerarquía permite capturar con precisión.

Catálogo

Excepciones comunes en .NET

Estas excepciones aparecen frecuentemente. Conocerlas evita confundirlas y permite capturarlas correctamente.

Tipo Descripción Ejemplo que la causa
Exception Clase base de todas las excepciones. No instanciar directamente. Usar clase derivada.
IndexOutOfRangeException El runtime la genera cuando una matriz no está correctamente indexada. arr[arr.Length + 1]
NullReferenceException Solo en runtime cuando se hace referencia a un objeto null. object o = null; o.ToString();
InvalidOperationException Los métodos la generan si se produce un estado no válido. Llamar a Enumerator.MoveNext() tras eliminar elemento de la colección.
ArgumentException Clase base para todas las excepciones de argumento. No instanciar directamente. Usar clase derivada.
ArgumentNullException Lanzada por métodos que no permiten argumentos nulos. string s = null; "texto".IndexOf(s);
ArgumentOutOfRangeException Lanzada cuando los argumentos están fuera del rango válido. string s = "hi"; s.Substring(s.Length + 1);
NullReferenceException e IndexOutOfRangeException son casi siempre bugs del programador: indican un error en el código, no en los datos del usuario.
Capturar

Capturar excepciones: try / catch / finally

Estructura básica y variantes para capturar excepciones específicas, múltiples tipos, y garantizar la liberación de recursos.

Ejemplos.cs
// ── 1. Captura básica ───────────────────────────────────────────
try
{
    int[] arr = new int[5];
    arr[10] = 1;                     // IndexOutOfRangeException
}
catch (IndexOutOfRangeException ex)
{
    Console.WriteLine($"Índice fuera de rango: {ex.Message}");
}

// ── 2. Múltiples catch (de más específico a más general) ────────
try
{
    string s = null;
    _ = s.Length;                    // NullReferenceException
}
catch (NullReferenceException ex)
{
    Console.WriteLine("Referencia nula.");
}
catch (Exception ex)
{
    Console.WriteLine($"Otro error: {ex.Message}");   // catch-all
}

// ── 3. Finally garantiza limpieza ───────────────────────────────
FileStream file = null;
try
{
    file = File.OpenRead("datos.txt");
    // … procesar …
}
catch (FileNotFoundException)
{
    Console.WriteLine("Archivo no encontrado.");
}
finally
{
    file?.Close();   // se ejecuta SIEMPRE
}

// ── 4. Captura sin variable (cuando no se usa ex) ───────────────
try { /* ... */ }
catch (ArgumentNullException) { /* no necesito ex */ }
Orden importa: siempre de la excepción más específica a la más general. catch (Exception) al final captura todo lo que no se capturó antes.
Lanzar

Lanzar excepciones: throw

Cómo crear y lanzar excepciones, cuándo relanzar, y cómo encadenar excepciones para no perder el contexto original.

LanzarEjemplos.cs
// ── 1. Lanzar una nueva excepción ───────────────────────────────
public void SetAge(int age)
{
    if (age < 0 || age > 150)
        throw new ArgumentOutOfRangeException(nameof(age),
            $"La edad {age} está fuera del rango válido (0-150).");
    this.age = age;
}

// ── 2. throw (sin argumento) — preserva el stack trace ──────────
try
{
    RiskyOperation();
}
catch (IOException ex)
{
    Log(ex);         // registrar
    throw;           // ← relanzar SIN resetear el stack trace
}

// ── 3. throw ex — PIERDE el stack trace (evitar) ────────────────
catch (IOException ex)
{
    throw ex;        // ← el stack trace ahora empieza aquí ⚠
}

// ── 4. Excepción encadenada (InnerException) ────────────────────
try
{
    File.ReadAllText("config.json");
}
catch (IOException ex)
{
    // Envuelve la excepción original para agregar contexto
    throw new ApplicationException(
        "No se pudo cargar la configuración.", ex);
    //                                         ↑ InnerException
}

// ── 5. Validación de argumentos (patrón guard clause) ───────────
public string Process(string input)
{
    ArgumentNullException.ThrowIfNull(input);          // .NET 6+
    ArgumentException.ThrowIfNullOrEmpty(input);       // .NET 7+
    // … lógica …
    return input.ToUpper();
}
Detalle crítico

throw vs throw ex: stack trace

Esta distinción es frecuente en errores de producción: perder el stack trace hace muy difícil encontrar el origen del error.

❌ throw ex (malo)
try
{
    // La excepción ORIGINÓ en la línea 42
    // de DatabaseHelper.ExecuteQuery()
    db.ExecuteQuery(sql);
}
catch (SqlException ex)
{
    Log(ex.Message);
    throw ex;   // ← Stack trace ahora dice:
                //   "line 8 of CatchBlock"
                //   El origen real se PERDIÓ
}
El stack trace queda apuntando al catch, no al lugar real del error.
✓ throw (correcto)
try
{
    // La excepción ORIGINÓ en la línea 42
    // de DatabaseHelper.ExecuteQuery()
    db.ExecuteQuery(sql);
}
catch (SqlException ex)
{
    Log(ex.Message);
    throw;      // ← Stack trace preservado:
                //   "line 42 of
                //    DatabaseHelper.ExecuteQuery"
}
El stack trace conserva el origen real. Mucho más útil para depurar.
Si necesitás agregar contexto al relanzar, usá throw new MiException("contexto", ex) donde ex es la InnerException. Así no perdés el error original.
¿Cuándo lanzar?

¿Cuándo lanzar vs. cuándo no?

Las excepciones son para situaciones excepcionales. Usarlas para control de flujo normal degrada la performance y la legibilidad.

SÍ lanzar cuando…
  • ✓
    Un método recibe un argumento inválido (null, fuera de rango).
  • ✓
    Una precondición no se cumple (el objeto está en estado incorrecto para esa operación).
  • ✓
    Una operación de I/O falla (archivo no encontrado, sin acceso, red caída).
  • ✓
    Se viola una invariante o postcondición de la especificación.
NO lanzar cuando…
  • ✗
    La condición es parte del flujo normal (ej: buscar algo que puede no existir → usar TryGet).
  • ✗
    El error se produce con altísima frecuencia (las excepciones tienen costo).
  • ✗
    Solo para saltar a otra parte del código (antipatrón de control de flujo).
  • ✗
    Para retornar valores vacíos: usar null, Option, o Result types.
Patrón TryXxx — alternativa a excepción en flujo normal
// ❌ Usar excepción para flujo normal (costoso)
try { int n = int.Parse(userInput); }
catch (FormatException) { /* input no era número */ }

// ✓ Patrón TryParse — pensado para el caso "puede fallar"
if (int.TryParse(userInput, out int n))
    Console.WriteLine($"Número: {n}");
else
    Console.WriteLine("Entrada inválida");
Propiedades

Propiedades de System.Exception

El objeto excepción lleva información útil para el diagnóstico. Conocer estas propiedades permite escribir mejores mensajes de error y logs.

Propiedades principales
Message
Descripción legible del error.
StackTrace
Secuencia de métodos en el momento del error.
InnerException
Excepción que causó esta (encadenado).
Source
Nombre del objeto o aplicación que causó el error.
HelpLink
URL con información adicional sobre el error.
InspectarExcepcion.cs
try
{
    throw new InvalidOperationException(
        "No se puede procesar.",
        new NullReferenceException("id es null"));
}
catch (Exception ex)
{
    Console.WriteLine($"Tipo:    {ex.GetType().Name}");
    Console.WriteLine($"Mensaje: {ex.Message}");
    Console.WriteLine($"Inner:   {ex.InnerException?.Message}");
    Console.WriteLine($"Stack:");
    Console.WriteLine(ex.StackTrace);

    // Recorrer la cadena de excepciones
    Exception inner = ex.InnerException;
    while (inner != null)
    {
        Console.WriteLine($" → {inner.Message}");
        inner = inner.InnerException;
    }
}
Resumen

Buenas prácticas · resumen

Hacer
  • ✓
    Usar throw; para relanzar y preservar el stack trace.
  • ✓
    Capturar excepciones específicas antes que la base Exception.
  • ✓
    Usar finally para liberar recursos siempre.
  • ✓
    Usar using / await using para IDisposable (es un finally automático).
  • ✓
    Incluir mensaje descriptivo y InnerException al crear excepciones.
  • ✓
    Usar TryXxx cuando el fallo es parte del flujo normal.
No hacer
  • ✗
    No usar throw ex; (pierde el stack trace).
  • ✗
    No silenciar con catch { } vacío.
  • ✗
    No lanzar Exception directamente: usar clases derivadas específicas.
  • ✗
    No usar excepciones para flujo de control normal.
  • ✗
    No capturar Exception sin relanzar (a menos que sea el punto de entrada).
  • ✗
    No crear excepciones propias sin los constructores estándar y [Serializable].

Las excepciones son para situaciones excepcionales. Úsalas para comunicar errores reales, no para controlar el flujo normal del programa.
