SOLID · Principio 2

Open /
Closed
Principle

Escribe clases que puedas extender sin tocar su código. El secreto de los sistemas que envejecen bien.

C# / .NET Bertrand Meyer · 1988 FIT – UCU

OCP es uno de los cinco principios SOLID de Robert C. Martin. Su origen está en el libro "Object Oriented Software Construction" de Bertrand Meyer (1988).

Motivación

¿Por qué necesitamos OCP?

Un programa se escribe una vez, pero se modifica muchas veces más. Cada modificación puede introducir errores donde antes todo funcionaba.

El riesgo habitual
  • !
    Para agregar una funcionalidad, modificamos clases que ya funcionan.
  • !
    Esas modificaciones pueden romper lo que antes estaba bien.
  • !
    Hay que re-testear todo el código afectado.
Lo que OCP propone
  • 1
    Diseñar abstracciones en las partes que se espera que cambien.
  • 2
    Extender el comportamiento via herencia o composición.
  • 3
    Nunca modificar el código existente que ya funciona.
La adhesión a OCP produce los mayores beneficios en POO: reusabilidad y mantenibilidad. Pero requiere pensar abstracciones, no solo usar un lenguaje OO.
Definición

El Enunciado

Las clases deben ser abiertas a la extensión,
pero cerradas a la modificación.

— Bertrand Meyer, 1988 · retomado por Robert C. Martin
Abierta a la extensión

Las responsabilidades de la clase pueden aumentarse mediante herencia, o mediante una combinación de herencia y composición/agregación.

Cerrada a la modificación

No es posible (ni necesario si se cumple el principio) realizar cambios en el código de esa clase. Lo que ya funciona, no se toca.

Contexto

El problema: agregar descuentos

El sistema de punto de venta tiene tickets con líneas de producto. Pero no soporta descuentos. Para agregarlos, habría que modificar clases existentes → viola OCP.

Una solución naïve sería usar un producto llamado "Descuento" con precio negativo, pero usar clases para propósitos no previstos es incorrecto y fragiliza el diseño.
Abstracción

La clave: clase abstracta SalesBaseItem

Introducimos SalesBaseItem, clase abstracta que define el contrato para cualquier línea de ticket. Cada subclase decide cómo calcular su subtotal y cómo imprimirse.

SalesBaseItem.cs
public abstract class SalesBaseItem
{
    // Subclase define cómo
    // calcular el subtotal
    public abstract double SubTotal { get; }

    // Subclase define qué
    // texto imprimir
    public abstract string GetTextToPrint();
}
Cualquier nueva "línea de ticket" (descuento, impuesto, recargo…) solo necesita heredar de SalesBaseItem e implementar sus métodos abstractos. El resto del sistema no cambia.
Implementación

SalesLineItem hereda de SalesBaseItem

La clase existente se adapta para extender SalesBaseItem, sobrescribiendo SubTotal y GetTextToPrint().

SalesLineItem.cs
public class SalesLineItem : SalesBaseItem   // hereda de la clase abstracta
{
    public SalesLineItem(double quantity, ProductSpecification product)
    {
        this.Quantity = quantity;
        this.Product = product;
    }

    public double Quantity { get; set; }
    public ProductSpecification Product { get; set; }

    public override double SubTotal     // implementa el contrato
    {
        get { return this.Quantity * this.Product.Price; }
    }

    public override string GetTextToPrint()
    {
        return $"{this.Quantity} de '{this.Product.Description}' a ${this.Product.Price}\n";
    }
}

La lógica de negocio de SalesLineItem permanece exactamente igual. Solo se agrega el : SalesBaseItem y los override.

Nueva clase

SaleDiscount: extensión sin modificar

Se agrega soporte a descuentos creando una nueva subclase. Ninguna clase existente se modifica.

SaleDiscount.cs
public class SaleDiscount : SalesBaseItem
{
    public SaleDiscount(double amount)
    {
        this.Amount = amount;
    }

    public double Amount { get; }

    public override double SubTotal
    {
        get
        {
            return -this.Amount;   // valor negativo → se resta del total
        }
    }

    public override string GetTextToPrint()
    {
        return $"Descuento: -${this.Amount}\n";
    }
}

SalesBaseItem no se modificó. Solo se extendió con una nueva subclase.

SubTotal retorna el monto en negativo para que se reste automáticamente del total.

Ajuste mínimo

Ajuste en Sale: List<SalesBaseItem>

El único cambio necesario en Sale es que la lista ahora almacena SalesBaseItem en lugar de SalesLineItem, y se agrega el método AddDiscount.

Sale.cs – cambios mínimos
public class Sale
{
    // Ahora almacena cualquier SalesBaseItem (línea, descuento, impuesto…)
    private List<SalesBaseItem> lineItems = new List<SalesBaseItem>();

    public SalesLineItem AddLineItem(double quantity, ProductSpecification product)
    {
        SalesLineItem item = new SalesLineItem(quantity, product);
        this.lineItems.Add(item);   // funciona igual que antes
        return item;
    }

    // Nuevo método — no modifica los anteriores
    public SaleDiscount AddDiscount(double amount)
    {
        SaleDiscount item = new SaleDiscount(amount);
        this.lineItems.Add(item);
        return item;
    }

    public double Total
    {
        get { return this.lineItems.Sum(i => i.SubTotal); }
    }
}
Agregar AddDiscount es un cambio aditivo: no modifica métodos existentes. Riesgo de introducir errores: mínimo. (El apéndice muestra cómo evitar incluso este cambio con inyección de dependencias.)
Resultado

El programa en acción

Con el nuevo diseño, agregar un descuento al ticket es trivial desde Program:

Program.cs
sale.AddLineItem(1, ProductAt(0));   // 1 × $100
sale.AddLineItem(2, ProductAt(1));   // 2 × $200
sale.AddLineItem(3, ProductAt(2));   // 3 × $300
sale.AddDiscount(50);                // descuento de $50
El total se calcula sumando los SubTotal de cada SalesBaseItem. El descuento resta automáticamente al devolver valor negativo. Sin modificar la lógica de cálculo de Sale.
Extensibilidad

Agregar SaleTax sin tocar nada

El diseño permite agregar un impuesto porcentual creando solo una nueva clase. El resto del sistema no se modifica.

SaleTax.cs – nueva extensión
public class SaleTax : SalesBaseItem
{
    private Sale sale;
    private double percentage;

    public SaleTax(double percentage, Sale sale)
    {
        this.percentage = percentage;
        this.sale = sale;
    }

    public override double SubTotal
    {
        get
        {
            // Calcula % sobre el subtotal
            // de las líneas sin impuesto
            return this.sale.SubtotalBeforeTax
                   * this.percentage / 100;
        }
    }

    public override string GetTextToPrint()
    {
        return $"IVA ({this.percentage}%): " +
               $"${this.SubTotal:F2}\n";
    }
}

Avanzado · Apéndice

Inyección de dependencias y delegados

Para evitar agregar un nuevo método Add* en Sale por cada nueva subclase, usamos inyección de dependencias con un delegado Func<SalesBaseItem>.

Sale.cs – un solo método Add universal
public class Sale
{
    // Un único método Add acepta cualquier creador de SalesBaseItem
    public SalesBaseItem Add(Func<SalesBaseItem> createItem)
    {
        SalesBaseItem item = createItem.Invoke();  // ejecuta el delegado
        this.lineItems.Add(item);
        return item;
    }
}
¿Qué es un delegado?

En C#, los métodos son objetos llamados delegados. Func<SalesBaseItem> representa una función sin parámetros que retorna un SalesBaseItem.

Lambda =>

Una función lambda se escribe con => (se lee "da como resultado"). Ej: x => x * x eleva al cuadrado. () => new SaleDiscount(50) crea un descuento.

Avanzado · Lambda

Funciones lambda en Program

Usando lambdas como argumento del método Add, la clase Sale ya no necesita un método específico por cada tipo de ítem:

Program.cs – con lambdas
// Cada lambda crea la instancia adecuada al momento de invocarla
sale.Add(() => new SalesLineItem(1, ProductAt(0)));   // línea 1
sale.Add(() => new SalesLineItem(2, ProductAt(1)));   // línea 2
sale.Add(() => new SalesLineItem(3, ProductAt(2)));   // línea 3
sale.Add(() => new SaleDiscount(50));                 // descuento
sale.Add(() => new SaleTax(10, sale));                // impuesto 10%

// () => ...  significa: "una función sin parámetros que da como resultado..."
Ahora podemos agregar cualquier nueva subclase de SalesBaseItem sin tocar ni una sola línea de Sale. OCP cumplido al 100%.
Nota para el curso: este apéndice cubre conceptos avanzados (delegados, lambdas, inyección de dependencias). No se espera que los apliques, pero es útil conocerlos como posibilidad del lenguaje.
Cierre

Resumen OCP

Abierto a extensión

Nuevas responsabilidades se agregan creando nuevas subclases de la clase abstracta.

Cerrado a modificación

El código que ya funciona no se toca, eliminando el riesgo de romper lo existente.

Clave: abstracciones

OCP no se consigue solo con un lenguaje OO. Requiere pensar qué puede cambiar y diseñar abstracciones ahí.

Beneficios

Reusabilidad y mantenibilidad: lo nuevo se agrega, lo viejo no se rompe. Sistemas que envejecen bien.

Extiende el comportamiento con nuevas clases. Nunca modifiques lo que ya funciona.
