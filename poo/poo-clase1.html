Expert Pattern SRP C#

// concepto base

¿Qué son las responsabilidades?

En POO, cada objeto tiene dos tipos de responsabilidades que deben distribuirse cuidadosamente entre las clases del sistema.

Responsabilidades de Conocer

El objeto conoce y gestiona su parte de los datos del sistema. Cada clase "sabe" ciertos atributos que le corresponden.

public int Age { get; set; } public string Name { get; set; }
Responsabilidades de Hacer

El objeto ejecuta comportamientos y acciones. Cada clase "hace" ciertas operaciones que le corresponden según los datos que posee.

public bool IsAdult() { return this.Age >= 18; }
Clave: La distribución de responsabilidades determina la calidad del diseño. Un mal reparto genera acoplamiento y código difícil de mantener.

// patrón expert

El patrón Expert

Problema

¿Cuál es el principio básico para asignar responsabilidades a las clases de un sistema?

Solución

Asignar la responsabilidad al experto en los datos, es decir, a la clase que tiene los datos necesarios para cumplirla.

Expert es uno de los patrones GRASP (General Responsibility Assignment Software Patterns) más fundamentales en el diseño OO.

// expert en acción

Expert: clase Person

Paso a paso, aplicamos Expert para asignar la responsabilidad de determinar si una persona es mayor de edad.

1
Identificar los datos necesarios

Para saber si alguien es mayor de edad necesitamos su edad.

2
Encontrar al experto en esos datos

La clase Person ya tiene la propiedad Age: es la experta.

3
Asignar la responsabilidad

Agregamos IsAdult() a la clase Person.

public class Person { // Responsabilidad de CONOCER public int Age { get; set; } // Responsabilidad de HACER // Expert: Person tiene Age ✔ public bool IsAdult() { return this.Age >= 18; } }
La responsabilidad queda donde están los datos. Esto reduce acoplamiento y mejora cohesión.

// principio srp

Principio de Responsabilidad Única

SRP

Single Responsibility Principle

"Nunca debería haber más de una razón para que una clase cambie."

Una clase con múltiples responsabilidades tiene múltiples razones para cambiar. Cada cambio en una responsabilidad puede afectar a las demás, generando fragilidad en el sistema.

¿Cuándo aplicarlo?
!
Cuando una clase mezcla lógica de negocio con persistencia
!
Cuando los cambios en una funcionalidad rompen otra
!
Cuando el algoritmo de cálculo puede variar independientemente de la entidad

// srp · análisis del problema

Person con sobrepeso: tensión

Agregamos peso y altura a Person. Ahora queremos calcular si tiene sobrepeso usando el IMC. ¿La lógica va en Person?

public class Person { public double Weight { get; set; } public double Height { get; set; } // ¿Debería ir aquí? public bool HasOverweight() { double imc = this.Height / Math.Pow(this.Weight, 2); return imc > 25; } }
Problema de diseño

Expert dice: los datos están en Person, entonces ahí va la lógica. Pero...

Si el criterio para determinar sobrepeso cambia (otro índice, otra fórmula), la clase Person debe cambiar por razones ajenas a la persona como entidad.

Esto viola el SRP: Person tendría dos razones para cambiar.

// srp · solución

Solución: OverweightCalculator

Extraemos la lógica de cálculo a una clase separada. Cada clase tiene ahora una única razón para cambiar.

// Person: solo datos de identidad public class Person { public double Weight { get; set; } public double Height { get; set; } // sin lógica IMC ✔ } // Nueva clase: algoritmo IMC public static class OverweightCalculator { public static bool HasOverweight( Person person) { double imc = person.Height / Math.Pow(person.Weight, 2); return imc > 25; } }
Beneficios del diseño
Person cambia solo si la entidad persona cambia (nuevos atributos, etc.)
OverweightCalculator cambia solo si el algoritmo IMC cambia
Podemos cambiar la fórmula sin tocar la clase Person
Fácil de testear por separado y reutilizar

// análisis comparativo

Expert vs SRP: tensión de principios

Ambos principios coexisten pero a veces generan tensión. Saber cuándo priorizar uno sobre el otro es clave.

Expert dice...

Asigna a quien tiene los datos.
Person tiene Weight/Height → HasOverweight va en Person.

VS
SRP dice...

No mezcles razones de cambio.
El algoritmo IMC es ajeno a la entidad → extrae a OverweightCalculator.

Regla práctica: Si la responsabilidad de hacer puede cambiar independientemente de la entidad, aplica SRP y extrae esa lógica. Expert es el punto de partida, SRP es el refinamiento.

// conceptos relacionados

Cohesión y Acoplamiento

Aplicar Expert y SRP correctamente mejora estos dos indicadores clave de calidad del diseño.

Alta Cohesión

Una clase con alta cohesión tiene responsabilidades estrechamente relacionadas. Cada método contribuye a un propósito central.

Person solo gestiona datos de persona → alta cohesión.
OverweightCalculator solo calcula IMC → alta cohesión.
Bajo Acoplamiento

Un acoplamiento bajo significa que los cambios en una clase no fuerzan cambios en otras. Las clases son independientes.

Sin SRP, cambiar el cálculo de IMC obliga a modificar Person, creando acoplamiento no deseado.
Objetivo de diseño: Maximizar la cohesión y minimizar el acoplamiento en cada clase del sistema.

SRP

Una sola razón para cambiar por clase.

Resultado

Alta cohesión + Bajo acoplamiento.

Código mantenible, testeable y extensible
