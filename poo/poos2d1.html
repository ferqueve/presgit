Abstracción 2 /

Abstracción

El arte de simplificar la complejidad
Definición

Una abstracción expresa las características esenciales de un objeto, que lo distinguen de todos los demás tipos de objetos, y que provee límites conceptuales claramente definidos, relativos a la perspectiva del usuario.

Pensá en Google Maps: tiene un mapa de tráfico, uno aéreo y uno de terreno. Ninguno tiene todos los detalles del mundo real, pero cada uno resalta los aspectos relevantes para su propósito. Eso es exactamente la abstracción en software.

¿Para qué sirve?

Para manejar la complejidad del mundo real, seleccionando sólo las propiedades relevantes para el problema que queremos resolver.

Clave

La misma entidad (ej. una persona) puede tener abstracciones diferentes según el sistema: una app de RRHH y una app médica abstraen cosas distintas de una persona.

  • El mundo real es complejo —como un fractal: cuanto más lo analizas, más detalle encontrás.
  • La abstracción filtra: nos quedamos con lo que importa y descartamos lo que no.
  • Es el fundamento del modelado orientado a objetos.
Abstracción 3 /

Abstracción en Código

Ejemplo: clase Person

De una persona nos podemos quedar con lo que importa para nuestro sistema. Aquí elegimos: nombre, apellido y la capacidad de saludar a otra persona.

// Abstracción de una persona para nuestro sistema
public class Person
{
    public string Name       { get; set; }  // ← característica esencial elegida
    public string FamilyName { get; set; }  // ← característica esencial elegida

    // Comportamiento relevante para nuestro modelo
    public void SayHiTo(Person person)
    {
        Console.WriteLine("Hola " + person.Name + ", soy " + this.Name);
    }
}
Lo que SÍ incluimos
  • Nombre y apellido
  • Capacidad de saludar
Lo que NO incluimos
  • Altura, color de ojos
  • Historia clínica, domicilio…
  • (no relevantes para este sistema)

Regla práctica: una buena abstracción incluye lo mínimo necesario para que los objetos puedan cumplir sus responsabilidades. Ni más, ni menos.

Encapsulación 4 /

Encapsulación

Ocultar los detalles que no importan
Definición

La encapsulación es el proceso de ocultar todos los detalles de un objeto que no contribuyen a sus características esenciales. Es sinónimo de escondimiento de información.

¿Qué oculta?

Los atributos privados y la implementación interna de los métodos. Otros objetos no saben cómo está implementado algo, solo saben qué pueden pedirle.

¿Qué expone?

Solo la interfaz pública: los métodos y propiedades que otros objetos necesitan usar. El "contrato" que la clase ofrece al mundo.

public class Person
{
    private string name;         // ← privado, nadie accede directamente
    private string familyName;   // ← privado

    public string Name {           // ← público: acceso controlado
        get { return this.name; }
        set { this.name = value; }
    }
}

Con el televisor usás el control remoto (interfaz pública) sin necesidad de saber nada sobre los circuitos internos (implementación encapsulada). Si el fabricante cambia los circuitos, seguís usando el mismo control.

Estado y Comportamiento 5 /

Programa OO · Estado · Comportamiento

Cómo vive un objeto
Programa Orientado a Objetos

En un programa OO los datos y la lógica están distribuidos en forma razonablemente equilibrada entre un conjunto de objetos que colaboran solicitándose y prestándose servicios mediante el envío de mensajes.

Estado

Los datos que un objeto conoce, almacenados en sus atributos. Cada dato conocido = un atributo.

Ejemplo: Name = "Luis", Age = 37

Comportamiento

Las acciones que un objeto puede realizar, implementadas en sus métodos.

Ejemplo: SayHiTo(), IsAdult()

public class Person
{
    // ── Estado ──
    public string Name       { get; set; }
    public string FamilyName { get; set; }
    public int    Age        { get; set; }

    // ── Comportamiento ──
    public bool IsAdult()  => this.Age >= 18;
    public void SayHiTo(Person p) => Console.WriteLine($"Hola {p.Name}, soy {this.Name}");
}

En un buen diseño OO ningún objeto carga solo con todos los datos y toda la lógica. La distribución equilibrada hace el sistema más mantenible y extensible.

Mensajes 6 /

Mensajes y Selectores

La única forma de comunicación entre objetos
Mensaje

Los objetos se comunican exclusivamente mediante mensajes. El emisor quiere consultar/cambiar el estado o activar cierto comportamiento del receptor.

Selector

El selector es el nombre del método que el emisor desea que el receptor ejecute. En lucho.SayHiTo(chino) el selector es SayHiTo.

Person lucho = new Person("Luis", "Suárez");
Person chino  = new Person("Sergio", "Rochet");

// Emisor: chino  |  Receptor: lucho  |  Selector: SayHiTo
chino.SayHiTo(lucho);   // → "Hola Luis, soy Sergio"

// El emisor no sabe CÓMO está implementado SayHiTo en lucho
// Solo sabe QUÉ pedirle → eso es encapsulación
Estado no accesible directamente

Para conocer un atributo de otro objeto, debés enviarle un mensaje. El receptor puede devolver un valor guardado o calcularlo — el emisor no sabe la diferencia.

this

Dentro de un método, la palabra clave this referencia al objeto que está ejecutando ese método en ese momento.

Clase e Identidad 7 /

Clase · Público/Privado · Identidad · Igualdad

Clase

Una clase es una plantilla o molde para un conjunto de objetos que comparten los mismos atributos, métodos, relaciones y semántica.
Un objeto es una instancia de una clase.

Público / Privado

public: accesible a cualquier objeto de cualquier clase.
private: accesible solo a objetos de la misma clase en que se define.

Igualdad

Dos objetos son iguales cuando son de la misma clase y tienen los mismos valores de atributos.
En C# se implementa sobrescribiendo Equals()

Identidad

Cada objeto tiene una existencia separada de los demás, aunque tenga los mismos atributos. Dos objetos iguales no son el mismo objeto.

Person chino   = new Person("Sergio", "Rochet");
Person arquero = new Person("Sergio", "Rochet");

Console.WriteLine(chino == arquero);          // false  ← distinta identidad
Console.WriteLine(chino.Equals(arquero));     // false  (sin sobrescribir)

Person goleador = chino;    // misma referencia
Console.WriteLine(chino == goleador);         // true   ← mismo objeto

Respuesta 9 /

Respuesta

Abstracción vs Encapsulación
Respuesta

a) Abstracción: decidir qué características esenciales incluir en la clase (balance, owner, withdraw()) es una decisión de abstracción. Estamos filtrando el mundo real y eligiendo lo relevante para nuestro sistema.

b) Encapsulación: hacer balance privado y controlarlo con métodos es encapsulación. Ocultamos el detalle de implementación y exponemos solo la interfaz necesaria.

Regla nemotécnica: la abstracción responde "¿qué existe?" y la encapsulación responde "¿quién puede ver qué?". Trabajan en conjunto pero son conceptos distintos.

public class BankAccount  // ← abstracción: elegimos estas características
{
    private decimal balance;  // ← encapsulación: ocultamos el saldo directo
    public string  Owner  { get; set; }

    public void Withdraw(decimal amount)  // ← interfaz pública controlada
    {
        if (amount <= balance) balance -= amount;
    }
}
Ciclo de Vida 10 /

Constructor y Ciclo de Vida

Nacer, vivir y desaparecer
Constructor

Método especial para crear e inicializar nuevas instancias de una clase. En C# tiene el mismo nombre que la clase. Si definimos uno propio, el predeterminado desaparece.

Ciclo de vida

Va desde que se asigna memoria al objeto (con new) hasta que esa memoria se libera cuando el garbage collector actúa.

public class Person
{
    // Constructor propio: obliga a tener nombre y apellido desde el inicio
    public Person(string name, string familyName)
    {
        this.Name       = name;
        this.FamilyName = familyName;
    }
    public string Name       { get; set; }
    public string FamilyName { get; set; }
}

// Uso: ya no se puede crear una Person sin nombre
Person lucho = new Person("Luis", "Suárez");
// new Person(); ← ERROR: el constructor predeterminado ya no existe
Destructor / Finalizador

En C# el destructor se llama ~Person(). Libera recursos (archivos, conexiones) que el objeto abrió en el constructor.

Garbage Collector

El runtime libera automáticamente la memoria de objetos cuando ninguna variable los referencia. En general es transparente para el programador.

Memoria 11 /

Stack y Heap

Dónde viven variables y objetos
Stack (Pila)

Almacena las variables locales de cada método y sus parámetros. Las variables existen solo mientras el método se ejecuta; al terminar el método, el espacio se libera automáticamente.

Heap (Montículo)

Almacena los objetos creados con new. Los objetos viven hasta que ninguna variable los referencia. El Garbage Collector los elimina.

Propiedades 14 /

Propiedades, Getters y Setters

Acceso controlado al estado

Las propiedades son una forma cómoda de declarar el estado. Internamente usan variables de instancia privadas. Los getters leen el valor; los setters lo escriben.

public class Person
{
    // Propiedad automática (getter y setter implícitos)
    public string Name { get; set; }

    // Propiedad calculada: solo getter, no tiene setter
    public string FullName
    {
        get { return this.Name + " " + this.FamilyName; }
    }

    // Variable de instancia + getter/setter explícitos
    private string familyName;
    public string FamilyName {
        get { return this.familyName; }
        set { this.familyName = value; }
    }
}
Solo getter (read-only)

Una propiedad sin setter no puede ser modificada desde afuera. FullName es calculado: combina Name y FamilyName.

null check

Antes de usar un objeto recibido como parámetro, siempre verificar if (person != null) para evitar excepciones de referencia nula.

Métodos 15 /

Métodos de Instancia vs Métodos de Clase

Método de instancia

Se ejecuta en el contexto de un objeto concreto. Tiene acceso a this y puede leer/modificar el estado de ese objeto.

lucho.SayHiTo(chino)

Método de clase (static)

Está asociado a la clase, no a ningún objeto. No tiene acceso a this ni puede acceder directamente al estado de instancias.

Person.Create("Luis")

public class Person
{
    // Método de instancia: necesita un objeto Person para ejecutarse
    public bool IsAdult() => this.Age >= 18;

    // Método de clase (static): se ejecuta sin objeto, para crear uno
    public static Person Create(string name, string fn)
        => new Person(name, fn);
}

// Uso:
Person p = Person.Create("Luis", "Suárez");  // método de clase
bool adult = p.IsAdult();                      // método de instancia

El constructor es el método de clase más importante: no puede ser de instancia porque el objeto todavía no existe cuando se lo llama. La fábrica crea el producto; el producto no se crea a sí mismo.

Expert 17 /

Patrón Expert

¿A quién asignar la responsabilidad?
Problema
¿Cuál es el principio básico para asignar responsabilidades?
Solución
Asignar la responsabilidad al experto en los datos: la clase que tiene los datos necesarios para cumplir esa responsabilidad.
public class Person
{
    public int Age { get; set; }

    // Expert: Person tiene Age → Person determina si es mayor de edad
    public bool IsAdult() => this.Age >= 18;
}

Analogía: si querés saber si un paciente tiene fiebre, le preguntás al médico que tiene la historia clínica, no al recepcionista. El que tiene los datos es el experto.

Ventaja

Los datos y la lógica que opera sobre ellos se mantienen juntos en la misma clase. El diseño queda cohesivo y fácil de mantener.

Pero no siempre solo Expert

Expert tiene un límite: hay casos en que sería aplicable pero va en contra de otros principios, como el SRP que veremos a continuación.

SRP 18 /

Principio de Responsabilidad Única (SRP)

Single Responsibility Principle
Enunciado
"Nunca debería haber más de una razón para que una clase cambie."
// ❌ MAL: Person sabe calcular IMC → si cambia el criterio, Person cambia
public class Person
{
    public double Weight { get; set; }
    public double Height { get; set; }
    public bool HasOverweight() { ... }  // viola SRP
}

// ✅ BIEN: separamos la responsabilidad de calcular el sobrepeso
public static class OverweightCalculator
{
    public static bool HasOverweight(Person person)
    {
        double imc = person.Height / Math.Pow(person.Weight, 2);
        return imc > 25;
    }
}
¿Cuándo aplica?

Cuando una responsabilidad podría cambiar por razones ajenas a lo que la clase representa. Aquí el criterio de sobrepeso (IMC, %grasa…) puede cambiar, pero Person no debería cambiar por eso.

Expert + SRP

Expert y SRP pueden complementarse: Expert asigna la responsabilidad al que tiene los datos, pero SRP impone que una clase no acumule responsabilidades con razones de cambio diferentes.

Respuesta

Expert vs SRP en ShoppingCart

Expert dice: ShoppingCart tiene los productos y precios → asignale la responsabilidad del impuesto a ella.

SRP cuestiona: las reglas de impuesto pueden cambiar independientemente de la lógica del carrito. Si el IVA sube de 22% a 24%, o si hay exenciones, ShoppingCart tendría que cambiar por algo que no le concierne.

Buena solución: crear una clase TaxCalculator que recibe el carrito y aplica las reglas fiscales. El carrito sabe de productos; el calculador sabe de impuestos.

public class TaxCalculator
{
    public decimal Calculate(ShoppingCart cart)
        => cart.Total * 0.22m;   // lógica impositiva aislada
}
