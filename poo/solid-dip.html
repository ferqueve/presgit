Contexto – Los Principios SOLID
Origen

El término SOLID es un acrónimo mnemónico introducido por Robert C. Martin en su documento "Design Principles and Design Patterns". Son cinco principios para diseños OO más comprensibles, flexibles y fáciles de mantener.

S
Single Responsibility

Una clase debe tener una única razón para cambiar.

O
Open / Closed

Abierto para extensión, cerrado para modificación.

L
Liskov Substitution

Los subtipos deben poder sustituir a sus supertipos.

I
Interface Segregation

Muchas interfaces específicas mejor que una general.

D
Dependency Inversion

Depender de abstracciones, no de clases concretas.

← Este documento

Martin identificó que una de las causas de los malos diseños son las interdependencias indeseadas entre clases, que producen rigidez, fragilidad e inmovilidad.

El Problema – Interdependencias Indeseadas
Dependencia directa entre clases concretas

Cuando una clase de alto nivel A depende directamente de una clase de bajo nivel B, surgen problemas graves que afectan la calidad del diseño.

Rigidez

El diseño es difícil de cambiar porque cada cambio afecta muchas partes del programa. Un cambio en B obliga a cambiar A.

Fragilidad

Al hacer un cambio, partes inesperadas del programa dejan de funcionar, incluso en responsabilidades que A no consume directamente.

Inmovilidad

Es muy difícil reutilizar clases en otro programa, porque no pueden "desenredarse" del programa actual sin arrastrar a B.

Dependencia directa (MAL)

Cambios en B rompen A · A no puede reutilizarse sin B

Enunciado del DIP
Regla 1 – Clases

"Las clases de alto nivel no deben depender de clases de bajo nivel; ambas deben depender de abstracciones."

Regla 2 – Abstracciones

"Las abstracciones no deben depender de detalles; los detalles deben depender de abstracciones."

Diseño con Inversión de Dependencias (BIEN)

A y B dependen de la abstracción · B puede cambiar sin afectar A

El principio no invierte la dirección de la dependencia; introduce una abstracción (interfaz o clase abstracta) de la que tanto A como B dependen, eliminando el vínculo directo entre ellas.

Beneficios de Depender de Abstracciones
Sin DIP — A depende de B directamente
  • Cambios en B obligan a cambiar A
  • Cambios en B, aunque A no los consuma, pueden romper A
  • No es posible reutilizar A sin B
  • Tests unitarios de A requieren una instancia real de B
Con DIP — A depende de Abstracción
  • B puede cambiar libremente sin afectar A, siempre que respete el contrato
  • Es más difícil que A deje de funcionar por cambios en B
  • A puede reutilizarse con cualquier clase que implemente la abstracción
  • Tests de A usan un mock/stub de la interfaz
Las abstracciones en C# son interfaces y clases abstractas

Tanto interface como abstract class pueden servir. Las interfaces son preferidas cuando no hay estado compartido.

Permite inyectar dependencias

DIP habilita el patrón Dependency Injection: en lugar de crear instancias dentro de la clase, se reciben por constructor o propiedad.

Facilita las pruebas unitarias

Al depender de una interfaz, podemos pasar objetos mock o stub durante los tests, sin necesidad de instanciar la clase concreta.

Ejemplo – Punto de Venta v1 (Antes)
Contexto

En la aplicación de punto de venta, ConsolePrinter y FilePrinter dependen directamente de la clase concreta Sale para obtener el texto del ticket. Esto viola DIP.

ConsolePrinter.cs — depende de Sale (MAL)
ConsolePrinter.csC#
public class ConsolePrinter : IPrinter
{
  // Depende de clase concreta Sale ✗
  public void PrintTicket(Sale sale)
  {
    Console.WriteLine(sale.GetTextToPrint());
  }
}
FilePrinter.csC#
public class FilePrinter : IPrinter
{
  // También depende de Sale ✗
  public void PrintTicket(Sale sale)
  {
    File.WriteAllText("Ticket.txt",
        sale.GetTextToPrint());
  }
}
Diagrama v1
Problemas

Si Sale cambia, también deben cambiar ConsolePrinter y FilePrinter. No podemos reutilizar los printers con otras clases de contenido.

Ejemplo – Introduciendo la Abstracción (v2)
Solución

Definimos la interfaz ISaleContent de la que dependerán tanto ConsolePrinter como Sale. Luego modificamos las firmas de los métodos y la clase Sale para que la implemente.

ISaleContent.cs — Nueva interfazC#
// Nueva abstracción
public interface ISaleContent
{
  string GetTextToPrint();
}
ConsolePrinter.cs (v2)C#
public class ConsolePrinter : IPrinter
{
  // Ahora recibe ISaleContent ✓
  public void PrintTicket(
      ISaleContent saleContent)
  {
    Console.WriteLine(
        saleContent.GetTextToPrint());
  }
}
Sale.cs (v2) — implementa ISaleContentC#
public class Sale : ISaleContent
{
  // Implementa el contrato de la interfaz
  public string GetTextToPrint()
  {
    string result =
        $"Fecha: {this.DateTime}\n";
    foreach (SalesLineItem item
             in this.lineItems)
    {
      result += item.GetTextToPrint();
    }
    result += $"Total: ${this.Total}";
    return result;
  }
}
Diagrama v2

Program no necesita modificarse en esta versión — los cambios quedaron limitados a ConsolePrinter, FilePrinter y Sale.

Ejemplo – Nuevos Formatos sin Tocar los Printers (v3)
Flexibilidad total

Podemos cambiar completamente cómo se imprime el ticket introduciendo nuevas clases de contenido, sin modificar ConsolePrinter, FilePrinter ni Sale.

PlainSaleContent
MarkdownSaleContent
Program.cs
PlainSaleContent.csC#
public class PlainSaleContent : ISaleContent
{
  private Sale sale;

  public PlainSaleContent(Sale sale)
  { this.sale = sale; }

  public string GetTextToPrint()
  {
    string result =
        $"Fecha: {this.sale.DateTime}\n";
    foreach (SalesLineItem item in this.sale)
    {
      result += item.GetTextToPrint();
    }
    result += $"Total: ${this.sale.Total}";
    return result;
  }
}

PlainSaleContent implementa ISaleContent y tiene una instancia de Sale para acceder al contenido de la venta. Retorna el texto en formato plano.

MarkdownSaleContent.csC#
public class MarkdownSaleContent : ISaleContent
{
  private Sale sale;

  public MarkdownSaleContent(Sale sale)
  { this.sale = sale; }

  public string GetTextToPrint()
  {
    string result =
        $"Fecha: {this.sale.DateTime}\n" +
        "Qty|Prod|Price|\n" +
        "|---|:---------:|-----:|\n";

    foreach (SalesLineItem item
             in this.sale)
    {
      result += $"|{item.Quantity}" +
          $"|{item.Product.Description}" +
          $"|${item.Product.Price}|\n";
    }
    result += $"**Total: ${this.sale.Total}**";
    return result;
  }
}

MarkdownSaleContent retorna el contenido en formato Markdown (tablas, negrita) sin modificar ninguno de los printers existentes.

Program.cs — Usando los nuevos contentC#
IPrinter consolePrinter = new ConsolePrinter();
IPrinter filePrinter   = new FilePrinter("Ticket.txt");

// v3: creamos el content y lo pasamos
ISaleContent saleContent =
    new PlainSaleContent(sale);

consolePrinter.PrintTicket(saleContent);
filePrinter.PrintTicket(saleContent);

// Mismo printer, diferente formato!
filePrinter   = new FilePrinter("Ticket.md");
saleContent   = new MarkdownSaleContent(sale);

consolePrinter.PrintTicket(saleContent);
filePrinter.PrintTicket(saleContent);

ConsolePrinter y FilePrinter no cambiaron. Solo se crearon nuevas clases de contenido. Esto es DIP en acción: extendemos el sistema sin modificar las clases existentes.

Resumen Visual – DIP Aplicado
Diagrama final v3 — Todo depende de ISaleContent
Resultado
  • ConsolePrinter y FilePrinter nunca cambiaron
  • Sale no necesitó modificarse en v3
  • Se agregaron dos formatos nuevos sin tocar nada existente
  • Cualquier nueva clase que implemente ISaleContent funcionará
Versiones del ejemplo
  • v1: Printers dependen de Sale directamente ✗
  • v2: Se introduce ISaleContent, Sale la implementa ✓
  • v3: Nuevas clases de contenido, sin cambiar los printers ✓✓

INotificationSender.csC#
public interface INotificationSender
{
  void Send(string to,
           string message);
}

// Alto nivel depende de la interfaz
public class NotificationService
{
  private INotificationSender _sender;

  public NotificationService(
      INotificationSender sender)
  { _sender = sender; }

  public void Notify(string to,
                        string msg)
      => _sender.Send(to, msg);
}

{
  string Format(ReportData data);
}

// Alto nivel: no importa el formato
public class ReportGenerator
{
  private IReportFormatter _fmt;

  public ReportGenerator(
      IReportFormatter formatter)
  { _fmt = formatter; }

  public void Generate(ReportData d)
  {
    string output = _fmt.Format(d);
    Console.WriteLine(output);
  }
}
